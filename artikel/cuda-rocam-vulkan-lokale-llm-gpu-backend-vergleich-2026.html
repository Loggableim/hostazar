🔥 Warum das GPU-Backend entscheidend ist

Wer 2026 lokale Large Language Models (LLMs) betreibt, kommt an einer Frage nicht vorbei: Auf welchem Backend laufen meine Modelle? Während NVIDIA seit Jahren den KI-Markt dominiert, haben AMD mit ROCm und die Open-Source-Community mit Vulkan mächtig aufgeholt. Sogar Intel mischt mit SYCL im Rennen mit.

Der Grund: Inferenz auf der GPU ist 10–50× schneller als auf der CPU. Ein Llama-3.1-70B Modell mit 4-Bit-Quantisierung braucht auf einer modernen CPU ~30–60 Sekunden pro Token-Gruppe – auf einer Mittelklasse-GPU sind es 5–15 Tokens pro Sekunde. Der Unterschied zwischen einem gut eingestellten und einem falschen GPU-Backend kann über Faktor 3 bei der Performance entscheiden.

Dieser Guide vergleicht alle vier relevanten GPU-Backends für lokale LLMs: CUDA (NVIDIA), ROCm (AMD), Vulkan (universell) und SYCL (Intel). Mit Benchmark-Zahlen, Hardware-Empfehlungen und konkreten Setup-Anleitungen.

🧩 Die vier GPU-Backends im Überblick

🟢 CUDA – Der unangefochtene König

CUDA (Compute Unified Device Architecture) ist NVIDIAs proprietäres Parallelrechner-Framework und seit über einem Jahrzehnt das Maß aller Dinge im GPU-Computing. Jedes relevante LLM-Framework unterstützt CUDA als erstes Backend:

  • llama.cpp: CUDA-Backend seit 2023, ständig optimiert
  • Ollama: Nutzt automatisch CUDA bei NVIDIA-GPUs
  • vLLM, LocalAI, LM Studio: Alle setzen auf CUDA als Primär-Backend
  • TensorRT-LLM: NVIDIAs optimierte Inferenz-Bibliothek – nur CUDA

Vorteile: Beste Performance, größte Modell-Kompatibilität, optimierte Kernel (FlashAttention, PageAttention), umfassende Dokumentation, riesige Community.
Nachteile: NVIDIA-Hardware erforderlich, proprietär, kein Open Source.

🔴 ROCm – AMDs Aufholjagd

ROCm (Radeon Open Compute) ist AMDs Open-Source-Plattform für GPU-Computing. 2026 hat ROCm einen beeindruckenden Reifegrad erreicht:

  • llama.cpp: ROCm-Backend (HIP) – nahe CUDA-Parität bei RDNA3-GPUs
  • Ollama: ROCm-Unterstützung seit v0.3, aktiviert per OLLAMA_HIP_VISIBLE_DEVICES
  • PyTorch: ROCm-Builds für Radeon-GPUs verfügbar

Vorteile: Open Source, kein Vendor-Lock-in, AMD-Hardware günstiger als NVIDIA-Äquivalente, kompatible HIP-API.
Nachteile: Geringere Abdeckung (nur RDNA2/3 und CDNA-Karten), weniger optimierte Kernel, Setup aufwändiger, Linux bevorzugt (Windows via HIP SDK eingeschränkt).

🟣 Vulkan – Der universelle Joker

Vulkan ist eine plattformunabhängige Grafik- und Compute-API (von der Khronos Group), die in den letzten Jahren als LLM-Backend enorm an Bedeutung gewonnen hat. Besonders llama.cpps Vulkan-Backend hat 2025/2026 große Sprünge gemacht:

  • llama.cpp: Vollständiges Vulkan-Backend, unterstützt NVIDIA, AMD, Intel, Apple Silicon
  • Kompatibel mit: Jeder GPU, die Vulkan 1.3 unterstützt (praktisch alle GPUs ab 2018)
  • Keine Treiber-Installation: Vulkan-Treiber sind meist vorinstalliert

Vorteile: Läuft auf jeder GPU (NVIDIA, AMD, Intel, Apple), kein proprietärer Vendor-Lock-in, einfaches Setup, Windows/Linux/macOS.
Nachteile: ~15–25% Performance-Einbuße gegenüber nativen CUDA/ROCm-Backends, weniger spezialisierte Kernel (z. B. kein FlashAttention), höherer Speicher-Overhead.

🔵 SYCL – Intels Ökosystem

SYCL ist eine plattformunabhängige C++-Programmierbibliothek für heterogenes Computing, die von Intel forciert wird. Über oneAPI bietet Intel ein komplettes KI-Toolkit:

  • llama.cpp: SYCL-Backend speziell für Intel Arc (Alchemist/Battlemage) und Intel Data Center GPUs
  • ipex-llm: Intels optimierte LLM-Bibliothek (ehemals BigDL-LLM)
  • Fokus: Intel Arc A-Serie, Intel Flex-Serie, integrierte Intel-Grafiken

Vorteile: Läuft auf Intel-GPUs (inkl. iGPUs!), Open Standard, Integration mit Intels oneAPI-Toolkit.
Nachteile: Kleine Community, wenig optimierte Kernel, Intel Arc erst ab Battlemage (BMG) konkurrenzfähig, Kompatibilitätsprobleme mit älteren Treibern.

⚙️ Setup-Guide: So richtest du die Backends ein

🟢 CUDA (NVIDIA) – Die Komplettlösung

# 1. CUDA Toolkit 12.x installieren
wget https://developer.download.nvidia.com/compute/cuda/repos/ubuntu2404/x86_64/cuda-keyring_1.1-1_all.deb
sudo dpkg -i cuda-keyring_1.1-1_all.deb
sudo apt update && sudo apt install cuda

# 2. llama.cpp mit CUDA bauen
git clone https://github.com/ggml-ai/llama.cpp
cd llama.cpp
cmake -B build -DGGML_CUDA=ON
cmake --build build --config Release -j$(nproc)

# 3. Test: GPU wird erkannt
./build/bin/llama-cli --help | grep -i cuda
# → "ggml_cuda_init: GGML_CUDA=1"
# → "ggml_cuda_init: found 1 CUDA devices: NVIDIA RTX 4090"

🔴 ROCm (AMD) – Die Open-Source-Alternative

# 1. ROCm 6.x installieren (Ubuntu)
sudo apt update
wget https://repo.radeon.com/amdgpu-install/6.2/ubuntu/jammy/amdgpu-install_6.2.60200-1_all.deb
sudo dpkg -i amdgpu-install_6.2.60200-1_all.deb
sudo amdgpu-install --usecase=rocm

# 2. llama.cpp mit HIP/ROCm bauen
cd llama.cpp
cmake -B build -DGGML_HIP=ON -DAMDGPU_TARGETS=gfx1100
cmake --build build --config Release -j$(nproc)

# 3. Aktivierung (Ollama)
export OLLAMA_HIP_VISIBLE_DEVICES=0
ollama serve

Wichtig: AMDGPU_TARGETS muss zu deiner GPU passen: gfx1030 (RX 6000), gfx1100 (RX 7900 XT), gfx1102 (RX 7600). Falsche Targets führen zu Kompilierungsfehlern oder Laufzeitabstürzen.

🟣 Vulkan – Der Universaltreiber

# 1. Vulkan SDK installieren (falls nicht vorhanden)
sudo apt install vulkan-tools libvulkan-dev

# 2. llama.cpp mit Vulkan bauen
cd llama.cpp
cmake -B build -DGGML_VULKAN=ON
cmake --build build --config Release -j$(nproc)

# 3. Prüfen
vulkaninfo --summary | grep "GPU"
# → "GPU id: 0 (NVIDIA GeForce RTX 4090)"
# → "GPU id: 1 (AMD Radeon RX 7900 XTX)"

# 4. Bestimmte GPU auswählen (bei Multi-GPU)
./build/bin/llama-cli -m model.gguf -ngl 99 --main-gpu 0

Tipp: Vulkan benötigt keine zusätzlichen SDKs für die GPU – die Treiber reichen. Ideal für schnelle Tests und Multi-GPU-Setups.

🔵 SYCL (Intel) – Für Intel-Grafikkarten

# 1. Intel oneAPI installieren
wget https://registrationcenter-download.intel.com/akdlm/IRC_NAS/7de3a0e8-.../l_BaseKit_p_2026.0.0.xxx_offline.sh
sudo sh ./l_BaseKit_p_2026.0.0.xxx_offline.sh

# 2. llama.cpp mit SYCL bauen
cd llama.cpp
cmake -B build -DGGML_SYCL=ON -DCMAKE_C_COMPILER=icx -DCMAKE_CXX_COMPILER=icpx
cmake --build build --config Release -j$(nproc)

# 3. Intel GPU auswählen
export ONEAPI_DEVICE_SELECTOR=level_zero:0
./build/bin/llama-cli -m model.gguf -ngl 99

Hinweis: SYCL ist 2026 vor allem für Intel Arc A7 und Battlemage-GPUs relevant. Für NVIDIA- oder AMD-Besitzer gibt es kaum Gründe, SYCL CUDA/ROCm vorzuziehen – die Performance ist niedriger und der Setup-Aufwand höher.

Takeaways:

  • CUDA bleibt die Messlatte – kein Backend erreicht die rohe Token-Rate einer nativen NVIDIA-Umgebung.
  • ROCm auf AMD RDNA3 (RX 7900 XTX) ist erschreckend nah dran: 135 t/s vs. 182 t/s auf der RTX 4090, aber die 7900 XTX kostet ~200 € weniger.
  • Vulkan liegt konsistent ~20% hinter dem nativen Backend. Der Vorteil: Es läuft problemlos auf allen GPUs – perfekt für heterogene Umgebungen.
  • SYCL macht nur auf Intel Arc Sinn. Mit Battlemage (B580) erreicht es solide 85 t/s – für den Preis (ca. 250 €) respektabel.

💡 Unsere klare Empfehlung

Blaupause 1 – "Ich will, dass es einfach funktioniert":
Kaufe eine NVIDIA RTX 4070 (12 GB) oder gebrauchte RTX 3090 (24 GB). Installiere CUDA. Fertig. Alles funktioniert out-of-the-box, alle Modelle laufen, du hast die beste Performance. Nachteil: NVIDIA-Aufpreis.

Blaupause 2 – "Ich habe AMD und will kein Geld verschenken":
Eine RX 7900 XTX (24 GB) unter Linux mit ROCm liefert ~90% der CUDA-Leistung einer RTX 4090 – für 30% weniger Geld. Der Haken: Unter Windows ist ROCm experimentell, du brauchst Linux oder dual Boot.

Blaupause 3 – "Budget-Build mit Intel":
Die Intel Arc B580 (12 GB, ~250 €) ist 2026 der Preiskracher. Mit SYCL erreichst du 85 t/s bei 8B-Modellen. Perfekt für Einsteiger, die lokale LLMs ausprobieren wollen, ohne gleich 1000 € auszugeben.

Blaupause 4 – "Ich will keine NVIDIA und kein AMD":
Vulkan ist dein Freund. Es läuft auf jeder GPU, egal ob du eine alte GTX 1060, eine Intel iGPU oder eine Radeon RX 580 hast. Die Performance ist ~20% niedriger, aber es funktioniert überall.

⚠️ Fallstricke & häufige Fehler

  • VRAM-Kapazität falsch eingeschätzt. Ein 70B-Modell in Q4_K_M braucht mindestens 40 GB VRAM – auf einer 24 GB Karte läuft nur Q2 oder Q3. Nutze VRAM-Rechner vor dem Kauf.
  • ROCm auf Windows. Nicht unmöglich, aber eine Qual. ROCm unter WSL2 läuft stabiler als nativer Windows-Treiber. Für Windows-Nutzer ohne GPU-Zwang: CUDA oder Vulkan.
  • Vulkan + Multi-GPU. llama.cpps Vulkan-Backend verteilt standardmäßig auf alle GPUs – das kann zu Sub-Optimierungen führen. Mit --main-gpu N die Primär-GPU setzen.
  • SYCL + alte Treiber. Intel-Treiber vor Q3/2025 haben teilweise cl_khr_fp16 nicht korrekt implementiert – führt zu NaN-Outputs. Treiber immer aktuell halten.
  • llama.cpp Build-Version. Ein alter Build (vor Mai 2026) hat kein FlashAttention für ROCm oder keine --no-mmap-Optimierung für Vulkan. Immer den aktuellen master-Branch bauen.

🔮 Ausblick: GPU-Backends 2027

Die Entwicklung bleibt rasant:

  • CUDA: Blackwell-GPUs (RTX 5090) mit 32 GB VRAM werden CUDA weiter als Goldstandard festigen. TensorRT-LLM wird zum Standard für lokale LLMs.
  • ROCm: AMD arbeitet an ROCm 7.0 mit voller Windows-Unterstützung und optimierten Attention-Kernels. Der Aufholprozess wird 2027 abgeschlossen sein.
  • Vulkan: Die Khronos Group treibt Vulkan 1.4 mit dedizierten Compute-Erweiterungen für KI-Workloads voran. FlashAttention für Vulkan ist in Entwicklung.
  • SYCL: Mit Intels Battlemage-Architektur und der nächsten oneAPI-Generation könnte SYCL eine ernsthafte Alternative werden – bleibt aber Nischen-Lösung.

Unser Fazit 2026: Wer heute einsteigt, kauft NVIDIA und nutzt CUDA. Wer AMD-Fan ist oder Geld sparen will, setzt auf ROCm unter Linux. Vulkan ist der Retter in der Not – für Bastler, Multiboot-Systeme und exotische Hardware. SYCL bleibt Spezialisten vorbehalten.
