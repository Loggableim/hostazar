KI & LLM 7. Juli 2026 · 12 Min Lesezeit

LLM lokal hosten 2026 – Komplettguide von der GPU bis zur API

Von der Hardware-Auswahl über die Software-Inbetriebnahme bis zur fertigen OpenAI-kompatiblen API: Der vollständige Leitfaden für deinen eigenen LLM-Server – für Einsteiger und Fortgeschrittene.

— Anzeige —

Google AdSense Platzhalter

Große Sprachmodelle (Large Language Models, LLMs) haben sich vom Nischen-Hobby zu einer ernstzunehmenden Produktivitäts-Technologie entwickelt. 2026 ist der Markt ausgereift: Open-Source-Modelle wie Llama 3, Mistral, Qwen 2.5 und DeepSeek V3 erreichen oder übertreffen kommerzielle Cloud-Angebote – und laufen auf handelsüblicher Hardware. Der entscheidende Vorteil des lokalen Hostens: Deine Daten verlassen nie dein Netzwerk, du zahlst keine API-Credits pro Token und hast volle Kontrolle über Modell, Konfiguration und Verfügbarkeit.

Dieser Guide führt dich Schritt für Schritt durch den gesamten Prozess – von der Frage, welche Grafikkarte du brauchst, über die Wahl der richtigen Software (Ollama, vLLM, Docker) bis hin zur Bereitstellung einer OpenAI-kompatiblen REST-API, die sich nahtlos in bestehende Tools und Workflows integrieren lässt. Wir gehen auf die drei häufigsten Anwendungsszenarien ein: den reinen Text-Chat-Betrieb, die Programmierung mit lokalen KI-Assistenten (Cursor, Continue.dev, Claude Code) und den Produktivbetrieb mit mehreren gleichzeitigen Nutzern.

🔑 Kurzfazit: Ein alltagstauglicher LLM-Server für 7B-14B-Modelle startet 2026 bei rund 1.500–2.500 Euro Hardware-Investition (gekaufte Grafikkarte) oder ab 0,50–1,50 Euro/Stunde in der Cloud (gemietete GPU). Für Einsteiger ist Ollama + Open WebUI die schnellste Lösung – vom Download bis zum ersten Prompt vergehen keine 15 Minuten. Wer niedrige Latenz und viele parallele Anfragen braucht, setzt auf vLLM mit Tensor-Parallelism auf mehreren GPUs.

1. Hardware-Grundlagen: Was braucht ein LLM wirklich?

Das Herzstück jedes lokalen LLM-Servers ist die Grafikkarte (GPU). Während CPUs theoretisch in der Lage sind, Sprachmodelle auszuführen, sind sie um den Faktor 10–50 langsamer als eine moderne GPU. Der Grund: LLMs bestehen aus hunderten Milliarden Matrix-Multiplikationen, die parallel auf Tausenden von CUDA-Kernen ablaufen – genau das, wofür GPUs optimiert sind.

Das wichtigste Kriterium ist der VRAM (Video-RAM). Die Faustregel: Ein 7B-Modell (7 Milliarden Parameter) in 4-Bit-Quantisierung benötigt etwa 4–5 GB VRAM. Ein 13B-Modell benötigt 8–10 GB, ein 70B-Modell etwa 40–45 GB. Eine Quantisierung senkt den Speicherbedarf drastisch, während die Qualität nur minimal leidet – 4-Bit-Quantisierung ist 2026 der absolute Standard für lokale Nutzung. Die zweite wichtige Kennzahl ist die Rechenleistung, gemessen in TFLOPS (FP16/CUDA Cores). Sie bestimmt, wie schnell Tokens generiert werden – gemessen in Tokens pro Sekunde (t/s).

Das reicht für Experimente und gelegentliche Chat-Nutzung völlig aus. Oder du mietest eine GPU in der Cloud für 1–2 Euro pro Stunde – perfekt zum Testen, bevor du kaufst.

CPU, RAM und Speicher

Neben der GPU sind Arbeitsspeicher (RAM) und Massenspeicher (SSD/NVMe) die zweite tragende Säule. Die CPU muss die Daten an die GPU verfüttern – hier sind moderne Mehrkern-Prozessoren (z. B. AMD Ryzen 7/9 oder Intel Core i7/i9 der aktuellen Generation) völlig ausreichend. Wichtiger ist der RAM: Jedes geladene Modell liegt zusätzlich im Systemspeicher, ebenso wie das Betriebssystem und die Inferenz-Software. Plane 32 GB RAM als Minimum ein – 64 GB sind komfortabel. Der Speicher sollte eine schnelle NVMe-SSD sein – Modelle mit 20 GB und mehr laden in Sekunden, alte HDDs brauchen Minuten.

2. Software-Stack: Die richtige Plattform wählen

Der Software-Stack für lokales LLM-Hosting ist 2026 erstaunlich ausgereift. Drei Optionen unterscheiden sich in Komplexität und Leistung:

Ollama – Der Einsteiger-König

Ollama ist die einfachste Möglichkeit, LLMs lokal auszuführen. Ein Befehl lädt ein Modell und startet einen REST-Server. Ollama unterstützt Dutzende Modelle (Llama 3.3, DeepSeek R1, Mistral, Qwen 2.5, Phi-4, Gemma 3), GPU-Beschleunigung via CUDA/ROCm und eine OpenAI-kompatible API für Tools wie Cursor oder Open WebUI.

# Ollama installieren (Linux, macOS, Windows via WSL2)
curl -fsSL https://ollama.com/install.sh | sh

# Modell herunterladen und starten
ollama pull llama3.3:70b-instruct-q4_K_M
ollama serve
# API läuft auf http://localhost:11434/v1

vLLM – Der Performance-Champion

vLLM ist die erste Wahl, wenn du maximale Durchsatzleistung benötigst – mehrere parallele Anfragen, Batch-Processing, Streaming und die fortschrittlichste Speicherverwaltung (PagedAttention). vLLM erreicht 10–15× höheren Durchsatz als naive Implementierungen und wird in Produktionsumgebungen bei Unternehmen wie OpenAI-kompatiblen APIs eingesetzt. Der Haken: vLLM benötigt CUDA und aktuell NVIDIA-GPUs (AMD-Unterstützung über ROCm ist experimentell). Die Einrichtung ist aufwändiger, aber die API ist vollständig OpenAI-kompatibel.

# vLLM via pip installieren
pip install vllm

# Server starten (z. B. mit DeepSeek R1 Distill Qwen 32B)
python -m vllm.entrypoints.openai.api_server \
  --model deepseek-ai/DeepSeek-R1-Distill-Qwen-32B \
  --tensor-parallel-size 1 \
  --gpu-memory-utilization 0.95 \
  --max-model-len 32768 \
  --port 8000

# API erreichbar unter http://localhost:8000/v1

Docker: Der universelle Rahmen

Sowohl Ollama als auch vLLM lassen sich hervorragend in Docker-Containern betreiben. Das hat gleich mehrere Vorteile: Isolation vom Host-System, einfache Updates per Container-Tausch, und reproduzierbare Setups via Docker Compose. Besonders für den Produktivbetrieb auf einem dedizierten Server ist Docker die sauberste Lösung. NVIDIA Container Toolkit (nvidia-container-toolkit) sorgt dafür, dass die GPU im Container durchgereicht wird.

# Ollama per Docker Compose mit Open WebUI
services:
  ollama:
    image: ollama/ollama:latest
    ports:
      - "11434:11434"
    volumes:
      - ./ollama:/root/.ollama
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: all
              capabilities: [gpu]

  open-webui:
    image: ghcr.io/open-webui/open-webui:main
    ports:
      - "3000:8080"
    environment:
      - OLLAMA_BASE_URL=http://ollama:11434
    volumes:
      - ./open-webui:/app/backend/data

⚙️ Setup-Tipp: Installiere zuerst das NVIDIA Container Toolkit (sudo apt install nvidia-container-toolkit und Neustart des Docker-Daemons). Ohne diesen Schritt erkennen Container die GPU nicht – ein typischer Anfängerfehler. Prüfe mit docker run --rm --gpus all nvidia/cuda:12.4.1-base-ubuntu22.04 nvidia-smi, ob alles funktioniert.

3. Die richtige Modell-Wahl 2026

Die Auswahl an Open-Source-Modellen ist 2026 so groß wie nie. Die wichtigsten Kategorien und Empfehlungen:

7B–8B Modelle (Einsteiger, CPU-kompatibel)

  • Llama 3.2 3B / Llama 3.1 8B – Meta's Allrounder. Exzellent in Reasoning und Befolgung von Anweisungen. 8B läuft auf 16 GB VRAM in Q4.
  • Qwen 2.5 7B – Alibabas Modell ist besonders stark in Mathematik und Code. Bester Open-Source-7B für technische Aufgaben.
  • Mistral 7B v0.3 – Der Klassiker. Immer noch konkurrenzfähig, besonders in englischsprachigen Anwendungen.
  • Phi-4 14B – Microsofts kompaktes Kraftpaket. Trotz nur 14B Parametern fast auf 70B-Niveau bei logischen Aufgaben.

30B–70B Modelle (Fortgeschritten, 24+ GB VRAM)

  • Llama 3.3 70B – Der Goldstandard. Bester Kompromiss aus Intelligenz und Hardware-Anforderungen. Läuft auf einer RTX 5090 (32 GB) in Q4.
  • DeepSeek R1 70B / DeepSeek V3 – Chinesisches Spitzenmodell mit herausragenden Reasoning-Fähigkeiten (ähnlich OpenAI o1/o3). Benötigt 48 GB VRAM für Q4.
  • Qwen 2.5 72B – Stark in multilingualen Anwendungen (inklusive Deutsch) und Code. Läuft auf 2× RTX 4090.
  • Mistral Large 2 (123B) – Nur für Multi-GPU-Setups geeignet, aber das leistungsfähigste Open-Source-Modell 2026.

Quantisierung verstehen

Quantisierung reduziert die Genauigkeit der Modell-Gewichte von 16-Bit-Fließkommazahlen auf 4-Bit- oder gar 2-Bit-Integer-Werte. Der Speicherbedarf sinkt auf ein Viertel oder Achtel, während die Qualität meist nur um 1–5 % abfällt. Die gängigsten Formate 2026 sind GGUF (für llama.cpp/Ollama) und AWQ/GPTQ (für vLLM). Empfehlung: Für Ollama nutze Q4_K_M – den besten Kompromiss. Für vLLM wähle AWQ, wenn verfügbar.

4. Schritt-für-Schritt: Der erste LLM-Server in 15 Minuten

So richtest du einen voll funktionsfähigen LLM-Server mit Ollama und Open WebUI in einer Viertelstunde ein:

  1. Ollama installieren: curl -fsSL https://ollama.com/install.sh | sh – das Skript erkennt deine GPU und installiert alles Nötige (CUDA-Treiber, Ollama-Binary, Systemd-Service).
  2. Modell herunterladen: ollama pull llama3.3:70b-instruct-q4_K_M – bei 70B ca. 40 GB Download. Alternativ für den Schnellstart: ollama pull llama3.2:3b – läuft auf jeder GPU mit 4 GB VRAM.
  3. Server starten: ollama serve – der REST-Server läuft auf Port 11434. Teste mit curl http://localhost:11434/v1/chat/completions -d '{"model":"llama3.3:70b-instruct-q4_K_M","messages":[{"role":"user","content":"Hallo!"}]}'
  4. Open WebUI starten: Mit Docker Compose (siehe oben) installierst du die schönste UI für Ollama – inklusive Chat-Verlauf, RAG-Files-Upload und Admin-Panel.
  5. Tools anbinden: Konfiguriere Cursor, Continue.dev (VS-Code-Plugin) oder deine eigene Anwendung auf http://deine-server-ip:11434/v1 als Custom OpenAI Endpoint.

5. API-Bereitstellung: Von lokal zu nutzbar

Der gesamte Sinn eines lokalen LLM-Servers liegt darin, ihn nutzbar zu machen. Sowohl Ollama als auch vLLM stellen eine OpenAI-kompatible REST-API bereit. Das bedeutet: Jedes Tool, das OpenAI unterstützt (und das sind 2026 quasi alle), kann auf deinen lokalen Server zugreifen – ohne Code-Änderungen. Einfach die Base-URL von https://api.openai.com/v1 auf http://localhost:11434/v1 (Ollama) oder http://localhost:8000/v1 (vLLM) ändern.

Sicherheitsaspekte bei der API-Freigabe

Solange du der einzige Nutzer bist, reicht localhost. Wenn du den Server im Netzwerk freigeben möchtest, beachte folgende Punkte:

  • Nicht direkt ins Internet hängen! Ein offener LLM-API-Endpoint ohne Authentifizierung ist ein Sicherheitsrisiko – Unbefugte könnten deine GPU-Ressourcen nutzen.
  • Reverse Proxy mit Auth: Setze Nginx oder Caddy als Reverse-Proxy davor, ergänzt um Basic Auth oder API-Keys. Oder nutze Cloudflare Tunnel mit Access-Regeln.
  • Firewall-Regeln: Beschränke den Zugriff auf dein LAN (z. B. 192.168.0.0/16) oder einzelne IPs.
  • vLLM API-Key: vLLM unterstützt API-Key-Authentifizierung über die Umgebungsvariable API_KEY.
# Beispiel: Caddy-Reverse-Proxy mit Basic Auth vor Ollama
# /etc/caddy/Caddyfile
llm.meinedomain.de {
  reverse_proxy localhost:11434
  basicauth {
    benutzer $2a$14$hashdespassworts
  }
}

6. Anwendungsfälle: Was machst du mit deinem LLM-Server?

Lokaler Chat mit Open WebUI

Die schönste UI für lokale LLMs ist Open WebUI mit ChatGPT-artigem Chat, RAG-Dokumentensuche, multimodaler Unterstützung und Benutzerverwaltung. Perfekt für Teams.

KI-gestützte Programmierung

Cursor und Continue.dev unterstützen benutzerdefinierte API-Endpunkte. Hinterlege deine lokale Ollama/vLLM-Adresse und dein Code-Editor nutzt deine GPU für Code-Vervollständigung und Refactoring – ideal, wenn du proprietären Code nicht an Cloud-APIs senden möchtest.

Autonome KI-Agenten

Frameworks wie LangChain, AutoGen oder CrewAI nutzen OpenAI-kompatible APIs. Mehrere KI-Agenten orchestrieren auf deinem lokalen Server Research, Content-Generierung oder Datenanalyse – ohne dass eine Anfrage dein Netzwerk verlässt.

7. Betrieb und Wartung

Ein LLM-Server erfordert regelmäßige Pflege:

  • Model-Updates: Neue Versionen erscheinen wöchentlich auf Hugging Face und GitHub – oft signifikant besser bei gleichem VRAM.
  • Software-Updates: ollama update oder docker pull ollama/ollama alle paar Wochen bringt Leistungsverbesserungen.
  • Kühlung: GPU unter Volllast wird heiß. Bei Multi-GPU-Systemen ist aktive Kühlung Pflicht. nvidia-smi zeigt Temperaturen an.
  • Stromkosten: Eine RTX 4090 unter Volllast zieht 350–450 Watt. Bei 24/7-Betrieb sind das etwa 80–120 €/Monat (0,30 €/kWh).

Entscheidungshilfe und nächste Schritte

Anfänger: Starte mit Ollama auf deinem PC – auch ohne GPU reicht CPU + 32 GB RAM für Llama 3.2 3B (~5–8 t/s). Deine nächste Investition: eine gebrauchte RTX 3090 (24 GB, ab 800 €) – das beste Preis-Leistungs-Verhältnis für LLMs 2026.

Fortgeschrittene: Baue einen Server mit RTX 5090 (32 GB) oder zwei RTX 4090. Setze vLLM mit Nginx-Reverse-Proxy und API-Key-Auth ein. Dein Team profitiert von schnellen, lokalen KI-Responses ohne API-Kosten.

Enterprise: Miete dedizierte GPU-Server bei Hetzner (A100/RTX 4090 ab 1,50 €/h). vLLM in Docker mit Kubernetes – deine Compliance-Abteilung wird es dir danken.

Egal, für welchen Weg du dich entscheidest: 2026 war noch nie ein besserer Zeitpunkt, um LLMs selbst zu hosten. Die Hardware ist erschwinglicher, die Software ausgereifter und die Modelle leistungsfähiger als je zuvor. Dein eigener KI-Server wartet.
