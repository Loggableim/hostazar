Multi-Model-Serving: Warum mehrere LLMs parallel?

Ein einzelnes Large Language Model ist wie ein Schweizer Taschenmesser – es kann vieles, aber für spezialisierte Aufgaben gibt es oft die bessere Wahl. Im Jahr 2026 hat sich gezeigt, dass Multi-Model-Serving der effektivste Weg ist, eine KI-Infrastruktur für Teams und Multi-Use-Cases zu betreiben. Statt ein monolithisches Modell für alle Anfragen zu verwenden, betreibst du mehrere spezialisierte LLMs parallel: ein leichtes Modell für einfache Chat-Anfragen, ein rechenstarkes Reasoning-Modell für komplexe Code-Analysen, ein deutsches Finetune für datenschutzkritische Dokumente und ein Vision-Modell für Bildverarbeitung – alles auf einem einzigen GPU-Server.

Die Kombination aus vLLM als hochperformantem Inference-Server und Open WebUI als benutzerfreundlicher Oberfläche macht genau das möglich: mehrere Modelle gleichzeitig bereitstellen, GPU-Speicher dynamisch teilen und per API-Routing die Anfragen intelligent verteilen. Dieser Guide zeigt dir, wie du eine solche Multi-Model-KI-Infrastruktur auf deinem eigenen VPS oder dedizierten Server aufbaust.

TL;DR – Das Wichtigste vorab

  • vLLM ist der aktuell schnellste Open-Source-Inference-Server (PagedAttention, Continuous Batching)
  • GPU-Memory-Sharing erlaubt mehrere Modelle auf einer GPU – via vLLM nativer Multi-Lora- und Multi-Model-Unterstützung
  • Model-Hot-Swapping ohne Server-Neustart über die vLLM-API
  • Open WebUI bietet eine einheitliche Chat-Oberfläche für alle Modelle
  • Empfohlene Hardware: NVIDIA A100 (80 GB) oder 2× RTX 4090 für 3–4 Modelle parallel

Was ist vLLM? – Der schnellste Open-Source-Inference-Server

vLLM ist ein von der UC Berkeley entwickeltes, quelloffenes Inferencing-Framework, das sich auf hohen Durchsatz und niedrige Latenz bei LLM-Inferenz spezialisiert hat. Das Herzstück ist der patentierte PagedAttention-Algorithmus: Statt den gesamten KV-Cache eines Requests als zusammenhängenden Speicherblock zu allozieren, teilt vLLM ihn in kleine Pages auf – genau wie ein Betriebssystem virtuellen Arbeitsspeicher verwaltet. Das Ergebnis ist eine drastische Reduzierung von Speicherfragmentierung und eine bis zu 2–4× höhere Durchsatzrate gegenüber herkömmlichen Lösungen wie Hugging Face Transformers oder Text Generation Inference (TGI).

Seit Version 0.6.x unterstützt vLLM auch Multi-Model-Serving nativ: Du startest einen einzigen vLLM-Server, der mehrere Modelle gleichzeitig lädt und über separate API-Endpunkte bereitstellt. Jedes Modell bekommt einen reservierten oder dynamischen Anteil des GPU-Speichers – der Server verteilt die Ressourcen automatisch nach Bedarf.

vLLM vs. Ollama – Wann setzt du welches Tool ein?

In unserem vorherigen Guide haben wir Ollama als LLM-Server auf dem VPS vorgestellt. Beide Tools sind exzellent, aber sie adressieren unterschiedliche Szenarien. Die folgende Vergleichstabelle hilft dir bei der Entscheidung:

Multi-Model-Serving ★ Nativ ab v0.6 – mehrere Modelle parallel ★ Nur sequenziell (ein Modell pro Server-Instanz)

Die einfache Faustregel: Einzelnutzer oder schnelle Experimente → Ollama. Produktiver Multi-Model-Betrieb im Team → vLLM.

Hardware-Anforderungen: GPU-Memory-Sharing und Multi-Model-Betrieb

Der größte Engpass beim Multi-Model-Serving ist der VRAM. Jedes LLM benötigt je nach Größe und Quantisierung zwischen 2 GB (Phi-3 Mini, Q4) und 70 GB (Llama 3.3 70B, FP16). Die Kunst liegt darin, mehrere Modelle so auf einer oder mehreren GPUs zu verteilen, dass sie sich den verfügbaren Speicher fair teilen.

vLLM unterstützt drei Strategien für Multi-GPU-Multi-Model:

  • Automatic GPU Selection: vLLM verteilt eingehende Requests automatisch auf verfügbare GPUs – Modelle teilen sich den Pool.
  • Pipeline-Parallelism: Ein großes Modell wird über mehrere GPUs aufgeteilt (Layer-Sharding). Ideal für 70B+ Modelle auf 2–4 GPUs.
  • Tensor-Parallelism: Noch feinere Verteilung für maximale Performance – jedes Tensor-Weight liegt auf mehreren GPUs.

Standard RTX 4090 24 GB 4–6 × 7B oder 1× 70B (Q4) 140–180 €/Monat

Tipp: GPU-Speicher clever verwalten

vLLM unterstützt —max-model-len und —gpu-memory-utilization (z. B. 0.85 für 85 %). Für Multi-Model-Betrieb startest du den Server mit —enable-multimodal und definierst pro Modell einen model_config-Eintrag mit eigener max_num_batched_tokens. So verhinderst du, dass ein einzelnes Modell den gesamten VRAM belegt.

Installation und Konfiguration von vLLM auf dem VPS

Wir gehen von einem frischen Ubuntu 24.04 LTS mit NVIDIA-Treibern und CUDA 12.4+ aus. Ist dein System noch nicht vorbereitet, lies zunächst unseren Guide Ollama LLM Server auf VPS hosten für die grundlegende CUDA- und Treiber-Installation – die Schritte 1–3 sind identisch.

Schritt 1: Python-Umgebung und vLLM installieren

vLLM läuft als Python-Paket. Wir empfehlen eine frische Conda-Umgebung oder venv:

sudo apt install -y python3-pip python3-venv git-lfs
python3 -m venv /opt/vllm-env
source /opt/vllm-env/bin/activate
pip install --upgrade pip
pip install vllm==0.6.3  # Aktuelle Stable-Version 2026

Schritt 2: Modelle herunterladen

vLLM arbeitet mit Modellen vom Hugging Face Hub. Lade sie vorab mit huggingface-cli oder git lfs:

# Modelle in /opt/models/ organisieren
mkdir -p /opt/models
cd /opt/models

# Kleines Chat-Modell (3.8B)
git lfs clone https://huggingface.co/microsoft/Phi-3-mini-4k-instruct

# Reasoning-Modell (7B)
git lfs clone https://huggingface.co/mistralai/Mistral-7B-Instruct-v0.3

# Code-Modell (14B, Q4 quantisiert)
git lfs clone https://huggingface.co/Qwen/Qwen2.5-Coder-14B-Instruct

# Deutsch-optimiertes Modell
git lfs clone https://huggingface.co/LeoLM/leo-mistral-7b-de

Schritt 3: Multi-Model-Server starten

Der entscheidende Schritt: Starte vLLM im Multi-Model-Modus mit einer Konfigurationsdatei. Erstelle /opt/vllm-config.yaml:

# /opt/vllm-config.yaml – Multi-Model-Serving Konfiguration
models:
  - model: /opt/models/Phi-3-mini-4k-instruct
    name: phi3-mini
    served_model_name: phi3-mini
    tokenizer_mode: auto
    max_model_len: 4096
    gpu_memory_utilization: 0.25  # 25 % des VRAM
    trust_remote_code: true

  - model: /opt/models/Mistral-7B-Instruct-v0.3
    name: mistral-7b
    served_model_name: mistral-7b
    max_model_len: 8192
    gpu_memory_utilization: 0.45

  - model: /opt/models/Qwen2.5-Coder-14B-Instruct
    name: qwen-coder
    served_model_name: qwen-coder
    max_model_len: 8192
    gpu_memory_utilization: 0.30

  - model: /opt/models/leo-mistral-7b-de
    name: leo-mistral
    served_model_name: leo-mistral-de
    max_model_len: 4096
    gpu_memory_utilization: 0.20

Starte den Server im Hintergrund:

source /opt/vllm-env/bin/activate
vllm serve --config /opt/vllm-config.yaml \
  --api-key mein-geheimer-key \
  --host 0.0.0.0 \
  --port 8000 \
  --enable-multimodal

vLLM startet nun alle vier Modelle parallel. Du erreichst sie unter:

# Jedes Modell via eigenem Endpunkt:
POST http://dein-server:8000/v1/chat/completions
{
  "model": "phi3-mini",
  "messages": [{"role": "user", "content": "Hallo!"}]
}

# Verfügbare Modelle auflisten:
GET http://dein-server:8000/v1/models

Schritt 4: Autostart mit systemd

Damit der Server nach einem Neustart automatisch läuft, erstelle eine systemd-Unit:

sudo cat > /etc/systemd/system/vllm.service << 'EOF'
[Unit]
Description=vLLM Multi-Model Inference Server
After=network.target

[Service]
Type=simple
User=root
Environment="CUDA_VISIBLE_DEVICES=0"
ExecStart=/bin/bash -c 'source /opt/vllm-env/bin/activate && vllm serve --config /opt/vllm-config.yaml --api-key mein-geheimer-key --host 0.0.0.0 --port 8000'
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable --now vllm

Open WebUI als einheitliches Frontend für alle Modelle

Open WebUI ist die ideale grafische Oberfläche für deinen Multi-Model-Server. Es bietet Chat-Verlauf, RAG (Retrieval Augmented Generation), Rollenverwaltung und vor allem: eine einheitliche Modellauswahl. Der Nutzer kann per Dropdown zwischen allen verfügbaren Modellen wechseln, ohne die API-URL zu ändern.

Die Installation per Docker ist unkompliziert – wichtig ist die Netzwerkkonfiguration, damit Open WebUI auf den vLLM-Server zugreifen kann:

docker run -d --network=host \
  -v open-webui-data:/app/backend/data \
  -e OPENAI_API_BASE_URL=http://localhost:8000/v1 \
  -e OPENAI_API_KEY=mein-geheimer-key \
  --name open-webui \
  ghcr.io/open-webui/open-webui:main

Rufe anschließend http://dein-server:8080 auf. Nach der Admin-Registrierung siehst du in der Modellauswahl alle vier konfigurierten Modelle – Phi-3 Mini für schnelle Chats, Mistral 7B für Allround-Fragen, Qwen Coder für Programmieraufgaben und Leo-Mistral DE für deutschsprachige Dokumente – und kannst nahtlos zwischen ihnen wechseln, ohne dass die laufenden Sessions unterbrochen werden.

Profi-Tipp: Modell-Routing per Prompt-Präfix

Nutze Open WebUIs Modell-Routing-Funktion: Definiere Regeln, die automatisch das passende Modell auswählen – z. B. /code als Präfix leitet an Qwen Coder weiter, /de an Leo-Mistral. Das spart deinem Team jedes Mal die manuelle Modellauswahl und sorgt für konsistente Ergebnisse.

Model-Hot-Swapping – Modelle wechseln ohne Neustart

Eines der stärksten Features von vLLM im Multi-Model-Betrieb ist das Hot-Swapping: Du kannst neue Modelle laden und alte entladen, ohne den Server zu stoppen. Die API-Endpunkte dafür sind:

# Neues Modell laden (ohne Neustart)
POST http://dein-server:8000/v1/models/load
{
  "model": "/opt/models/Llama-3.3-70B-Instruct",
  "name": "llama-70b",
  "gpu_memory_utilization": 0.50
}

# Modell entladen und Speicher freigeben
POST http://dein-server:8000/v1/models/unload
{
  "model": "phi3-mini"
}

# Aktuell geladene Modelle prüfen
GET http://dein-server:8000/v1/models

Beim Entladen eines Modells wird dessen VRAM-Anteil sofort freigegeben und steht anderen Modellen zur Verfügung – in Echtzeit, ohne Downtime.

API-Routing – Intelligente Verteilung der Anfragen

Sobald du mehrere Modelle parallel betreibst, stellt sich die Frage: Welches Modell bekommt welche Anfrage? Hier kommen API-Gateway-Strategien ins Spiel. Drei praxisbewährte Ansätze:

1. Client-seitiges Routing

Der Client (oder Open WebUI) wählt das Modell explizit aus. Einfach, aber der Nutzer muss wissen, welches Modell für welche Aufgabe geeignet ist.

2. Proxy-basiertes Routing mit OpenRouter oder LiteLLM

Dienste wie LiteLLM oder OpenRouter können als zentraler Proxy zwischen deinen Clients und deinen vLLM-Modellen geschaltet werden. LiteLLM erlaubt z. B. ein config.yaml mit Routing-Regeln:

# litellm-config.yaml
model_list:
  - model_name: "schnell"       # Freitext-Alias
    litellm_params:
      model: "openai/phi3-mini"
      api_base: "http://localhost:8000/v1"
      api_key: "mein-geheimer-key"
  - model_name: "profi"
    litellm_params:
      model: "openai/qwen-coder"
      api_base: "http://localhost:8000/v1"
  - model_name: "deutsch"
    litellm_params:
      model: "openai/leo-mistral-de"
      api_base: "http://localhost:8000/v1"

# Fallback-Regeln
router_settings:
  routing_strategy: "usage"      # oder "latency", "cooldown"
  enable_loadbalancing: true
  num_retries: 3

Deine Clients rufen dann einfach /v1/chat/completions mit "model": "schnell" auf – LiteLLM routet automatisch. Fällt ein Modell aus, wird auf das nächste passende Fallback-Modell gewechselt.

3. Content-basiertes Routing

Fortgeschrittene Setups nutzen ein kleines Router-Modell (z. B. Phi-3 Mini), das den Prompt analysiert und entscheidet, welches spezialisierte Modell die Anfrage bekommt – Code-Fragen → Qwen Coder, Deutsch → Leo-Mistral, Mathematik → DeepSeek R1. Dieses Setup implementierst du mit einem einfachen Python-Skript vor deinem vLLM-Server:

from openai import OpenAI

client = OpenAI(base_url="http://localhost:8000/v1", api_key="...")

def route_request(user_message: str) -> str:
    """Klassifiziert die Anfrage und wählt das passende Modell."""
    classifier_prompt = f"Klassifiziere die Anfrage als: 'code', 'chat', 'mathe' oder 'deutsch'. Antwort: Nur das Wort.\\n\\n{user_message}"
    response = client.chat.completions.create(
        model="phi3-mini",
        messages=[{"role": "user", "content": classifier_prompt}],
        max_tokens=10
    )
    category = response.choices[0].message.content.strip().lower()
    model_map = {"code": "qwen-coder", "chat": "mistral-7b",
                 "mathe": "deepseek-r1-7b", "deutsch": "leo-mistral-de"}
    return model_map.get(category, "mistral-7b")

Monitoring und Logging des Multi-Model-Servers

Ein Produktivsystem braucht Überwachung. vLLM liefert über den Endpunkt /metrics Prometheus-kompatible Metriken:

  • vllm:num_requests_running – Aktuell laufende Requests pro Modell
  • vllm:gpu_cache_usage – GPU-Speicher-Auslastung in Pages
  • vllm:avg_generation_throughput – Generierte Tokens pro Sekunde
  • vllm:request_latency_seconds – Latenz pro Request (Histogramm)

Diese Werte kannst du mit Prometheus und Grafana visualisieren, um Engpässe frühzeitig zu erkennen. Ein Alarm bei gpu_cache_usage > 0.90 warnt dich, bevor der Server Requests ablehnen muss. Nutze außerdem das Dashboard von Open WebUI, das in der Admin-Oberfläche eine detaillierte Nutzungsstatistik pro Modell anzeigt.

Sicherheit: Reverse Proxy und Authentifizierung

Wie bei jedem selbstgehosteten Server gilt: Setze niemals Port 8000 direkt ins Internet. Konfiguriere stattdessen einen Reverse Proxy – wir empfehlen Nginx oder Traefik – mit Let's Encrypt und API-Key-Authentifizierung:

# Nginx-Konfiguration (Ausschnitt)
server {
    listen 443 ssl;
    server_name ki.dein-server.de;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Authorization "Bearer mein-geheimer-key";
        proxy_set_header Host $host;
    }
}

Noch sicherer ist der Zugriff ausschließlich über WireGuard VPN für dich und dein Team. So bleibt die gesamte Kommunikation verschlüsselt und von außen unsichtbar.

Kostenvergleich: Multi-Model-Self-Hosting vs. Cloud-APIs

Lohnt sich der Betrieb mehrerer Modelle auf eigener Hardware? Wir vergleichen die monatlichen Kosten für ein typisches Team-Setup (4 Modelle, ~15 Millionen Tokens/Monat, 3 Nutzer):

  • OpenAI (GPT-4o + o3-mini + GPT-4.1-nano): ~320 $/Monat bei gemischter Nutzung. Keine Vorabkosten, aber keine Datenhoheit und keine Modell-Auswahl bei Spezialmodellen.
  • Anthropic Claude (Sonnet + Haiku): ~250 $/Monat. Exzellente Qualität, aber ebenfalls keine Datenkontrolle und nur zwei Modelle.
  • vLLM Self-Hosting (RTX 4090, 24 GB): Einmalig ~2.000 € für die GPU (Miete: ~150 €/Monat) + ~40 € VPS mit CPU und 64 GB RAM + ~30 € Strom. Gesamt: ~220 €/Monat für unbegrenzte Tokens, 4 Modelle und volle Datenhoheit.
  • vLLM Dedicated (Hetzner, 2× A100 80 GB): ~1.500 €/Monat für Enterprise-Teams mit hohem Durchsatz und bis zu 10 Modellen parallel.

Fazit: Ab etwa 3 aktiven Nutzern und 10 Millionen Tokens/Monat ist Self-Hosting mit vLLM die wirtschaftlichere Wahl – und du behältst die Kontrolle über deine Daten und Modellauswahl.

FAQ – Häufige Fragen zum Multi-Model-Server mit vLLM

1. Kann ich Ollama-Modelle auch in vLLM verwenden?

vLLM lädt Modelle direkt vom Hugging Face Hub oder aus lokalen Verzeichnissen. Ollama-Modelle (GGUF-Format) sind nicht direkt kompatibel – du musst die Hugging Face-Version des gleichen Modells herunterladen. Alternativ betreibst du beide Server parallel: Ollama für schnelle Experimente, vLLM für den Produktivbetrieb.

2. Wie viele Modelle laufen auf einer RTX 4090?

Mit 24 GB VRAM und Q4-Quantisierung passen 5–6 Modelle der Größe 7B (je ~4–5 GB) oder 2 Modelle der Größe 13–14B (je ~8–10 GB). Ein 70B-Modell in Q4 (~40 GB) passt nicht auf eine einzelne Karte – dann brauchst du Tensor-Parallelism über mehrere GPUs.

3. Was passiert, wenn ein Modell den gesamten VRAM belegt?

vLLM alloziert Speicher entsprechend der gpu_memory_utilization pro Modell. Überschreitet ein Request diese Grenze, wird er mit einem 503 Service Unavailable abgewiesen, statt den Server zum Absturz zu bringen. Überwache die Metrik vllm:gpu_cache_usage, um frühzeitig zu skalieren.

4. Unterstützt vLLM Vision- und Multimodal-Modelle?

Ja – seit Version 0.6 unterstützt vLLM Multimodal-Modelle wie LlaVA-NeXT, InternVL2 und Qwen2-VL. Du musst die Modelle wie gewohnt laden und aktivierst multimodale Inferenz über den Serve-Flag —enable-multimodal. Open WebUI zeigt Bilder dann direkt im Chat an.

5. Wie update ich die Modelle auf neuere Versionen?

Neue Modellversionen lädst du mit git pull im jeweiligen Modellverzeichnis oder via Hugging Face Hub. Nach dem Update entlädt du das alte Modell via API (/v1/models/unload) und lädst es neu (/v1/models/load) – kein Server-Neustart nötig.

6. Kann ich vLLM mit Docker betreiben?

Ja, das offizielle Image vllm/vllm-openai:latest unterstützt GPU-Passthrough (—gpus all). Für Multi-Model-Serving musst du die Konfigurationsdatei per Volume mounten: docker run —gpus all -v $(pwd)/config.yaml:/app/config.yaml .... Beachte, dass Docker den Shared Memory (—shm-size) oft erhöhen muss – —shm-size 32g ist ein guter Startwert.

7. Wie hoch ist die Latenz beim Modellwechsel (Hot-Swap)?

Das Laden eines 7B-Modells von NVMe-SSD dauert etwa 2–5 Sekunden, eines 70B-Modells 15–30 Sekunden. Während des Ladens laufen andere Modelle uneingeschränkt weiter. Entladen ist instantan – der Speicher wird sofort freigegeben.

8. Welche Quantisierung empfiehlt sich für Multi-Model-Serving?

Für Multi-Model-Betrieb ist AWQ (Activation-Aware Weight Quantization) die beste Wahl – sie bietet Q4-Qualität bei nahezu FP16-Performance und ist vLLM-nativ unterstützt. Lade AWQ-quantisierte Modelle direkt von Hugging Face (erkennbar am Label „AWQ“) – sie benötigen bis zu 60 % weniger VRAM als FP16-Varianten bei minimalem Qualitätsverlust.

Fazit

Multi-Model-Serving mit vLLM und Open WebUI hebt die selbstgehostete KI-Infrastruktur auf ein neues Level. Statt dich auf ein einzelnes Modell zu beschränken, betreibst du ein ganzes Ensemble spezialisierter LLMs parallel – ohne dass deine GPU-Budget explodiert. Dank PagedAttention, dynamischem GPU-Memory-Sharing und Model-Hot-Swapping von vLLM legst du einen hochperformanten, produktionsreifen KI-Server auf, der mit kommerziellen Cloud-APIs in puncto Geschwindigkeit und Flexibilität mithalten kann.

Die Kombination aus Open WebUI als nutzerfreundlichem Frontend, LiteLLM oder Eigenbau-Routing für intelligente Anfrageverteilung und Prometheus/Grafana für das Monitoring ergibt eine rundum professionelle Plattform – ob für dein Development-Team, deine Data-Science-Abteilung oder als SaaS-Angebot für interne KI-Assistenten. Wer datenschutzkonforme, leistungsstarke und vielseitige KI selbst hosten will, kommt an vLLM im Multi-Model-Setup nicht vorbei.
