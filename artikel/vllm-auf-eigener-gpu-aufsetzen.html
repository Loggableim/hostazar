vLLM auf eigener GPU aufsetzen – Einleitung

Large Language Models wie Llama 3.3 70B, Mixtral 8x22B oder Qwen 2.5 72B sind 2026 die treibende Kraft hinter zahllosen KI-Anwendungen – von Chatbots über Code-Assistenten bis hin zu Retrieval-Augmented-Generation-Systemen. Doch mit zunehmender Modellgröße wird die Inferenz zum Engpass: Selbst eine NVIDIA RTX 4090 mit 24 GB VRAM kommt bei einem 70B-Modell ohne Optimierung schnell an ihre Grenzen. Hier setzt vLLM an – ein hochoptimiertes Open-Source-Inference-Framework, das speziell für Production-Ready LLM Serving auf eigener GPU entwickelt wurde.

In diesem Guide zeigen wir dir, wie du vLLM auf deinem GPU-Server installierst, konfigurierst und für den produktiven Betrieb optimierst. Du erfährst, wie PagedAttention den VRAM-Verbrauch drastisch reduziert, wie Continuous Batching den Durchsatz maximiert und wie du mit Multi-LoRA Dutzende feinabgestimmte Modelle parallel bereitstellst – alles hinter einer OpenAI-kompatiblen REST-API. Inklusive Vergleichstabelle, Docker-Setup, Kubernetes-Deployment und Monitoring.

Was ist vLLM und warum brauchst du es?

vLLM ist ein von der UC Berkeley entwickeltes Open-Source-Inference-Framework (Apache 2.0), das speziell für die effiziente Ausführung großer Sprachmodelle auf GPUs optimiert wurde. Im Kern löst vLLM das zentrale Problem des LLM-Servings: Die KV-Cache-Verwaltung. Bei herkömmlichen Ansätzen wird der Attention-Key-Value-Cache jeder Anfrage als zusammenhängender Speicherblock im VRAM abgelegt – ein ineffizientes Verfahren, das zu massiver Speicherverschwendung (bis zu 60–80 %) führt und damit die Anzahl gleichzeitiger Anfragen drastisch begrenzt.

Die wichtigsten Features und Vorteile von vLLM:

  • PagedAttention: Der KV-Cache wird in kleine, nicht-zusammenhängende Blöcke (Pages) aufgeteilt – analog zum Paging-Betriebssystem-Prinzip. Ergebnis: nahezu 100 % VRAM-Auslastung und bis zu 2–4× höherer Durchsatz gegenüber Standard-Implementierungen.
  • Continuous Batching: Anders als bei statischem Batching werden Anfragen dynamisch im laufenden Betrieb zur Batch-Verarbeitung hinzugefügt oder entfernt – sobald eine Anfrage fertig inferiert wird, rückt die nächste nach. Das maximiert die GPU-Auslastung und senkt die Latenz pro Request.
  • OpenAI-kompatible API: Der vLLM-Server stellt eine REST-API bereit, die 1:1 zum chat/completions- und completions-Endpunkt von OpenAI kompatibel ist. Bestehende Tools, Libraries und Agenten (LangChain, LlamaIndex, Continue.dev, Open WebUI) können ohne Code-Änderung angebunden werden.
  • Multi-LoRA: Dutzende oder hunderte feinabgestimmte LoRA-Adapter können parallel geladen und dynamisch je nach Anfrage ausgewählt werden – ohne Modell-Switching oder redundante VRAM-Belegung.
  • Tensor Parallel & Pipeline Parallel: Große Modelle lassen sich transparent über mehrere GPUs und sogar mehrere Nodes verteilen.

Zum Vergleich: Während Ollama eher auf einfache, lokale Nutzung ausgelegt ist, zielt vLLM auf produktiven Multi-User-Betrieb mit hohem Durchsatz ab – der ideale Unterbau für SaaS-KI-Dienste, Unternehmenslösungen oder KI-gestützte Developer-Tools auf eigener GPU-Infrastruktur.

Die Wahl der richtigen GPU ist der entscheidende Faktor für die Performance deines vLLM-Setups. Die folgende Tabelle zeigt die GPU-Empfehlungen für die gängigsten Modellgrößen:

Modell Parameter Min. VRAM (Q4) Empfohlene GPU Max. Tokens/s
Llama 3.2 3B / Phi-3 Mini 3–4B 4 GB RTX 3060 / A10 ~180
Mistral 7B / Llama 3.1 8B 7–8B 8 GB RTX 4070 / A10 ~120
Qwen 2.5 14B / Phi-3 Medium 14B 14 GB RTX 4080 / A100 40GB ~80
Llama 3.3 70B / Mixtral 8x22B 70B 40 GB A100 80GB / 2× RTX 4090 ~45
Qwen 2.5 72B / DeepSeek V3 72B+ 48 GB A100 80GB / H100 ~35

CPU und RAM: Neben der GPU sind ein moderner Multi-Core-Prozessor (AMD EPYC oder Intel Xeon, mindestens 8 Kerne) sowie 32–128 GB System-RAM empfehlenswert – der CPU wandelt Token-Embeddings um und steuert den Datenfluss zur GPU. Für die SSD gilt: Eine schnelle NVMe mit 1 TB oder mehr, da Modelle in Q4-Quantisierung schnell 40–80 GB belegen.

Installation: vLLM auf dem GPU-Server einrichten

Wir zeigen dir drei Wege: die native pip-Installation (maximale Flexibilität), Docker (saubere Isolation) und das Kubernetes-Helm-Deployment (produktiver Standard). Als Betriebssystem empfehlen wir Ubuntu 24.04 LTS mit NVIDIA-Treiber 555+ und CUDA 12.4+.

Variante 1: Native Installation mit pip (Python 3.11+)

Die einfachste Methode für einen dedizierten GPU-Server:

# Voraussetzungen prüfen
nvidia-smi  # Sollte CUDA 12.4+ anzeigen
python3 --version  # 3.11 oder 3.12 empfohlen

# Python-venv einrichten
python3 -m venv vllm-env
source vllm-env/bin/activate

# vLLM installieren (pip-Version ~0.8.x Stand 2026)
pip install vllm

# Schnelltest: Server starten
vllm serve mistralai/Mistral-7B-Instruct-v0.3 \
  --tensor-parallel-size 1 \
  --gpu-memory-utilization 0.90 \
  --max-model-len 8192

vLLM lädt das Modell automatisch von Hugging Face herunter (sofern es öffentlich ist). Bei eigenem Speicherpfad setzt du --model /pfad/zum/Modell. Der Server startet standardmäßig auf http://0.0.0.0:8000 und stellt die OpenAI-kompatible REST-API bereit.

Variante 2: Docker (empfohlen für Produktion)

Das offizielle Docker-Image vllm/vllm-openai enthält alle Abhängigkeiten vorkonfiguriert und lässt sich mit GPU-Passthrough betreiben:

# vLLM mit Docker starten (GPU erforderlich)
docker run --gpus all \
  -e HF_TOKEN=${HF_TOKEN:-} \
  -p 8000:8000 \
  -v ~/.cache/huggingface:/root/.cache/huggingface \
  vllm/vllm-openai:latest \
  --model mistralai/Mistral-7B-Instruct-v0.3 \
  --max-model-len 8192 \
  --gpu-memory-utilization 0.90

# Test: Chat-Completion abrufen
curl http://localhost:8000/v1/chat/completions \
  -d '{"model":"mistralai/Mistral-7B-Instruct-v0.3","messages":[{"role":"user","content":"Erkläre PagedAttention in 3 Sätzen."}]}'

Für Modelle mit eingeschränktem Zugriff (z. B. Llama 3.3, Gemma 2) übergibst du den Hugging-Face-Token per HF_TOKEN-Umgebungsvariable – und stellst sicher, dass du die jeweilige Lizenz auf huggingface.co akzeptiert hast. Ein vollständiges docker-compose.yml für vLLM + Open WebUI + Nginx Reverse Proxy findest du in unserem Docker-Compose-VPS-Guide.

Variante 3: Kubernetes mit Helm (für Cluster-Betrieb)

Für skalierbare Produktionsumgebungen steht ein offizielles Helm-Chart bereit:

# Helm-Repository hinzufügen
helm repo add vllm https://vllm-project.github.io/helm-charts
helm repo update

# vLLM-Deployment installieren
helm install vllm vllm/vllm \
  --set model=mistralai/Mistral-7B-Instruct-v0.3 \
  --set tensorParallelSize=1 \
  --set gpuMemoryUtilization=0.9 \
  --set resources.limits.nvidia.com/gpu=1

Das Chart erstellt automatisch ein Deployment, einen Service (ClusterIP) und optional einen Ingress. Kombiniert mit dem K3s-Leightweight-Kubernetes-Setup und Nginx Reverse Proxy erhältst du eine vollständige LLM-Serving-Plattform.

PagedAttention und Continuous Batching – Die Innovationen im Detail

Um zu verstehen, warum vLLM so viel effizienter ist, lohnt ein Blick unter die Haube.

PagedAttention: Das Betriebssystem-Prinzip für den KV-Cache

Bei der Textgenerierung mit einem Transformer-Modell wird für jedes Token ein Key-Value-Cache (KV-Cache) angelegt – im Grunde eine Matrix aller Attention-Scores zwischen den bereits generierten Tokens. Bei einem 70B-Modell und einer Kontextlänge von 4096 Tokens belegt dieser Cache schnell 1–2 GB pro Anfrage. Das Problem: Herkömmliche Frameworks alloziieren diesen Cache als zusammenhängenden Speicherblock (contiguous memory), was zu Fragmentierung führt – selbst bei 80 GB VRAM sind oft nur 20–30 gleichzeitige Anfragen möglich.

vLLMs PagedAttention unterteilt den KV-Cache in feste Blöcke (Pages) von 16–32 Tokens. Diese Blöcke werden nicht-zusammenhängend im VRAM abgelegt – genau wie ein Betriebssystem Speicherseiten verwaltet. Die Attention-Berechnung erfolgt blockweise, und ein Block-Table (analog zur Page-Table) hält die Zuordnung zwischen logischen und physischen Blöcken. Ergebnis: Nahezu 100 % VRAM-Auslastung und bis zu 4× mehr gleichzeitige Anfragen bei gleichem VRAM.

⚡ Performance-Vergleich: vLLM mit PagedAttention

In Benchmarks mit Llama 3.3 70B auf einer A100-80GB erreicht vLLM 45–55 Tokens/s bei Q4-Quantisierung – das ist 3–5× schneller als ein naives Hugging-Face-Setup und 2× schneller als Text Generation Inference (TGI) bei vergleichbarem VRAM-Verbrauch. Die Speichereffizienz durch PagedAttention ermöglicht zudem 70–80 parallele Anfragen auf einer einzelnen A100 – TGI schafft hier typischerweise 20–30.

Continuous Batching: Volle GPU-Auslastung rund um die Uhr

Stell dir vor, dein LLM-Server bekommt 100 Anfragen pro Minute, jede mit unterschiedlicher Antwortlänge (von 50 bis 2000 Tokens). Ohne Continuous Batching müsstest du entweder statische Batches bilden (warte, bis N Anfragen da sind, verarbeite sie gemeinsam – erhöht die Latenz) oder jede Anfrage einzeln abarbeiten (schlechte GPU-Auslastung).

vLLMs Continuous Batching (manchmal auch in-flight batching genannt) löst dieses Problem elegant: Sobald eine Anfrage in der Batch- Verarbeitung ihr End-of-Sequence-Token erreicht hat, wird sie aus dem Batch entfernt und eine neue Anfrage rückt sofort nach. Die GPU muss nie warten – sie arbeitet kontinuierlich. In der Praxis bedeutet das: 70–90 % GPU-Auslastung selbst bei stark schwankendem Anfrageaufkommen, statt der typischen 20–40 % bei statischen Verfahren.

Aktiviert ist Continuous Batching in vLLM standardmäßig. Über die Parameter --max-num-seqs (maximale Anzahl paralleler Sequenzen) und --max-num-batched-tokens (maximale Anzahl Tokens pro Batch) kannst du das Verhalten an deine Hardware anpassen. Ein guter Startwert: --max-num-seqs 256 auf einer A100, --max-num-seqs 64 auf einer RTX 4090.

OpenAI-kompatible API einrichten und nutzen

Einer der größten Vorteile von vLLM ist die native OpenAI-kompatible REST-API. Der Server stellt zwei Endpunkte bereit – /v1/chat/completions für Chat-Modelle und /v1/completions für klassische Text-Complete-Modelle. Die API ist 100 % Drop-In-kompatibel: Du änderst lediglich die Base-URL in deinem Client.

# Python-Beispiel: OpenAI-Client gegen vLLM
from openai import OpenAI

client = OpenAI(
    base_url="http://dein-server:8000/v1",
    api_key="not-needed"  # vLLM hat keine Auth – Proxy macht das
)

response = client.chat.completions.create(
    model="mistralai/Mistral-7B-Instruct-v0.3",
    messages=[
        {"role": "system", "content": "Du bist ein deutscher DevOps-Experte."},
        {"role": "user", "content": "Wie konfiguriere ich vLLM für hohen Durchsatz?"}
    ],
    max_tokens=1024,
    temperature=0.3
)
print(response.choices[0].message.content)

Wichtig: Der vLLM-Server hat standardmäßig keine Authentifizierung. Setze daher zwingend einen Reverse Proxy (Nginx, Caddy oder Traefik) mit Basic-Auth, API-Key oder JWT davor. Nutze am besten ein WireGuard-VPN für den internen Zugriff – siehe unseren WireGuard-VPN-Guide. Eine umfassende Absicherung des gesamten Servers findest du in Linux-Server härten – Security-Baseline 2026.

Multi-LoRA: Dutzende Modelle auf einer GPU

Ein echtes Killer-Feature von vLLM ist die Multi-LoRA-Unterstützung. LoRA (Low-Rank Adaptation) ist eine Parameter-efﬁziente Fine-Tuning-Methode, bei der nur kleine Adapter-Matrizen trainiert werden, während die Basis-Gewichte des Modells eingefroren bleiben. Ein typischer LoRA-Adapter ist nur 5–50 MB groß – selbst bei hundert verschiedenen Adaptern liegt der zusätzliche Speicherbedarf unter 5 GB.

Mit vLLMs Multi-LoRA kannst du:

  • Einen Basis-Modell-Weight-Sharing-Pool betreiben: Ein Llama 3.3 70B wird einmal geladen (40 GB VRAM), dazu kommen 50 verschiedene LoRA-Adapter für verschiedene Aufgaben (Code, Kundenservice, deutscher Chatbot, SQL-Generator) – zusätzlicher VRAM: ~250 MB.
  • Den Adapter dynamisch pro Request auswählen: Im API-Request übergibst du den Parameter lora_name oder lora_request – vLLM lädt den entsprechenden Adapter im laufenden Betrieb.
  • Adapter heißen laden (hot-swap): Neue Adapter können über die API hinzugefügt werden, ohne den Server neuzustarten.
# Multi-LoRA starten
vllm serve mistralai/Mistral-7B-Instruct-v0.3 \
  --enable-lora \
  --lora-modules devops-advisor=/path/to/devops-lora \
  --max-lora-rank 64

# Per API einen bestimmten LoRA-Adapter anfragen
curl http://localhost:8000/v1/chat/completions \
  -d '{
    "model": "mistralai/Mistral-7B-Instruct-v0.3",
    "messages": [{"role":"user","content":"Gib mir einen Docker-Compose-Tipp."}],
    "lora_name": "devops-advisor"
  }'

Monitoring und Metriken: vLLM überwachen

Für den produktiven Betrieb ist ein Monitoring unerlässlich. vLLM bringt ein eingebautes Prometheus-Metriken-Exportformat mit, das du über den Endpunkt /metrics abrufen kannst (standardmäßig auf Port 8264). Die wichtigsten Metriken:

  • vllm:num_requests_running – aktuell laufende Anfragen
  • vllm:num_requests_waiting – Anfragen in der Warteschlange
  • vllm:gpu_cache_usage_pct – GPU-Cache-Auslastung in Prozent
  • vllm:prompt_tokens_total – insgesamt verarbeitete Prompt-Tokens
  • vllm:generation_tokens_total – insgesamt generierte Tokens
  • vllm:request_success_total – erfolgreiche Requests (HTTP 200)
  • vllm:request_latency_seconds – Latenz-Histogramm (p50, p95, p99)

Diese Metriken integrierst du nahtlos in dein bestehendes Prometheus & Grafana Monitoring. Ein fertiges Grafana-Dashboard für vLLM findest du auf grafana.com (Suche nach "vLLM").

🔧 Produktions-Tipp: vLLM mit Systemd als Dienst

Für den 24/7-Betrieb richtest du vLLM am besten als Systemd-Dienst ein. Lege eine Datei /etc/systemd/system/vllm.service an, die den Docker-Container oder das Python-Skript startet, mit Restart=always und RestartSec=5. Kombiniere dies mit logrotate für die Logs und einem Healthcheck via curl http://localhost:8000/health – vLLM liefert auf dem Health-Endpunkt HTTP 200, sobald das Modell geladen und bereit ist.

Skalierung: Tensor Parallel und Pipeline Parallel im Detail

Sobald ein Modell nicht mehr in den VRAM einer einzelnen GPU passt (bei Llama 3.3 70B in Q4 sind das ~40 GB – eine RTX 4090 hat 24 GB), musst du auf Multi-GPU-Verteilung setzen. vLLM unterstützt zwei komplementäre Verfahren:

Tensor Parallel (TP)

Bei TP werden die einzelnen Transformer-Layer horizontal auf mehrere GPUs aufgeteilt: Jede GPU hält einen Teil der Gewichtsmatrizen und berechnet nur ihren Anteil. Die GPU-Kommunikation erfolgt über NCCL (NVLink oder InfiniBand). TP reduziert die Latenz pro Token, erfordert aber schnelle GPU-Verbindungen. Ein Beispiel für 2 GPUs:

# Tensor Parallel über 2 GPUs
vllm serve mistralai/Mixtral-8x22B-Instruct-v0.1 \
  --tensor-parallel-size 2 \
  --dtype auto

Pipeline Parallel (PP)

PP teilt das Modell vertikal auf: GPU 1 berechnet die Layer 1–20, GPU 2 die Layer 21–40. Jede GPU hält die vollständigen Gewichte ihres Abschnitts. PP skaliert besser über langsamere Verbindungen (Ethernet) und eignet sich für Multi-Node-Setups. Der Durchsatz skaliert nahezu linear mit der Anzahl der GPUs:

# Pipeline Parallel über 2 GPUs + Tensor Parallel pro Node
vllm serve meta-llama/Llama-3.3-70B-Instruct \
  --tensor-parallel-size 2 \
  --pipeline-parallel-size 2 \
  --max-model-len 16384

Kostenbetrachtung: Lohnt sich eine eigene GPU für vLLM?

Die Antwort hängt von deinem Nutzungsprofil ab. Wir vergleichen drei Szenarien für 10 Millionen Tokens pro Monat (etwa 20.000 Chat-Anfragen à ~500 Tokens):

  • OpenAI GPT-4o (API): ~120–150 $ pro Monat bei gemischter Input/Output-Rate. Keine Vorabkosten, aber keine Datenhoheit.
  • Hetzner GPU Cloud (A100 80 GB): ~2,50 €/Stunde. Bei 8 Stunden/Tag (Spitzenlastabdeckung) ~250 €/Monat. Flexibel, aber teuer bei Dauerbetrieb.
  • Eigene GPU (RTX 4090 + CPU-Server): Einmalig ~2.000 € für die GPU, ~80 €/Monat für einen Hetzner-Dedicated-Server mit GPU-Slot. Amortisation nach ~14 Monaten gegenüber der API.
  • Dedizierter GPU-Server (2× A100): ~600–900 €/Monat bei OVHcloud oder Hetzner. Maximaler Durchsatz, Enterprise-Niveau.

Unser Tipp: Starte mit einer einzelnen RTX 4090 (oder warte auf die kommende RTX 5090 mit 32 GB VRAM) auf einem Hetzner-Dedicated-Server. Das Setup kostet dich rund 150–200 €/Monat all-inclusive und serviert problemlos 10–20 Millionen Tokens – weit unter dem API-Preis.

Fazit

vLLM ist 2026 der Goldstandard für Production-Ready LLM Serving auf eigener GPU-Infrastruktur. Mit PagedAttention löst es das VRAM-Fragmentierungsproblem, mit Continuous Batching maximiert es die GPU-Auslastung und mit der OpenAI-kompatiblen API integrierst du es nahtlos in bestehende Tools und Workflows. Egal ob du einen einzelnen Chatbot für dein Team betreiben, eine KI-gestützte IDE mit Code-Vervollständigung ausstatten oder einen skalierbaren B2B-LLM-Service aufbauen willst – vLLM liefert die Performance und Zuverlässigkeit, die für den produktiven Einsatz nötig ist.

Die Kombination aus vLLM und einem leistungsstarken GPU-Server ersetzt teure Cloud-API-Abonnements und gibt dir die volle Kontrolle über deine Daten, Modelle und Kosten zurück. Zusammen mit Tools wie Open WebUI als Frontend, Prometheus & Grafana für Monitoring und einem Nginx Reverse Proxy für Sicherheit baust du dir damit eine KI-Plattform, die mit kommerziellen Anbietern Schritt hält – zu einem Bruchteil der Kosten.
