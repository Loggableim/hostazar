KI-Server-Sicherheit: LLM API absichern, Rate-Limiting & Monitoring 2026

Security-Best-Practices für selbst gehostete Large Language Models – von API-Key-Auth bis DDoS-Schutz

— Anzeige —

Google AdSense Platzhalter

Warum Sicherheit bei selbst gehosteten LLMs kritisch ist

Immer mehr Unternehmen und Entwickler setzen 2026 auf selbst gehostete Large Language Models (LLMs) – sei es mit Ollama auf dem VPS, vLLM, Text Generation Inference (TGI) oder lokalen Setup mit llama.cpp. Der Vorteil liegt auf der Hand: volle Datenhoheit, keine externen API-Kosten und DSGVO-konforme Verarbeitung sensibler Informationen. Doch mit der Kontrolle über die Infrastruktur kommt auch die Verantwortung für deren Sicherheit.

Eine ungeschützte LLM-API ist ein offenes Einfallstor: Ohne Authentifizierung kann jeder mit einem curl-Befehl auf dein Modell zugreifen, Inference-Kosten verursachen, sensible Daten extrahieren oder sogar Prompt-Injection-Angriffe durchführen. In diesem Artikel zeigen wir dir, wie du deine KI-Server mit API-Key-Auth, Rate-Limiting, Request-Logging, Modell-Isolation, CORS-Richtlinien und DDoS-Schutz absicherst – praxisnah mit Nginx, Cloudflare und modernen Security-Tools.

1. API-Key-Authentifizierung – Die erste Verteidigungslinie

Jede LLM-API, die über das Netzwerk erreichbar ist, benötigt eine Authentifizierung. Die meisten selbst gehosteten LLM-Lösungen (Ollama, vLLM, LocalAI) liefern standardmäßig keine eingebaute Auth – sie binden den Server auf localhost und erwarten, dass du einen Reverse Proxy mit Authentifizierung vorschaltest.

1.1 Nginx mit API-Key-Auth

Die einfachste Methode: Nginx prüft einen statischen API-Key im Authorization-Header oder als Query-Parameter. So konfigurierst du einen geschützten Upstream für deine Ollama-API:

# /etc/nginx/sites-available/llm-api
server {
    listen 443 ssl;
    server_name llm-api.deinedomain.de;

    ssl_certificate /etc/letsencrypt/live/llm-api/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/llm-api/privkey.pem;

    location / {
        # API-Key prüfen
        if ($http_authorization !~ "^Bearer sk-[A-Za-z0-9]{32,}$") {
            return 401 '{"error":"Unauthorized – gültiger API-Key erforderlich"}';
        }

        proxy_pass http://127.0.0.1:11434;

        # Rate Limiting: max 10 req/s
        limit_req zone=llm burst=20 nodelay;

        # CORS-Header
        add_header Access-Control-Allow-Origin "https://meine-webui.de";
        add_header Access-Control-Allow-Headers "Authorization, Content-Type";
    }
}

Diese Konfiguration prüft einen Bearer-Token im Format sk-xxxxxxxx. Für produktive Umgebungen empfehlen wir eine dynamische Key-Validierung über eine kleine Lua-Engine (OpenResty) oder eine vorgeschaltete API-Gateway-Lösung wie Kong, Traefik oder KrakenD.

1.2 Multi-User-Auth mit API-Gateway

Für Teams und Organisationen reicht ein einzelner statischer Key nicht aus. Hier empfiehlt sich ein API-Gateway, das individuelle Keys verwaltet, rotiert und protokolliert:

  • Kong Gateway – Open-Source, Plugin-Ökosystem (Key-Auth, JWT, OAuth2, Rate-Limiting out-of-the-box)
  • Traefik – Docker-nativ, Middleware-Chain für Basic-Auth, Digest-Auth, Forward-Auth mit Authelia/Authentik
  • Nginx + OpenResty + Lua – Für Eigenbau-Lösungen mit Redis-Backend zur Key-Validierung
  • Cloudflare Access – Zero-Trust-Netzwerkzugriff ohne öffentliche IP, inklusive SSO (Google, GitHub, Azure AD)

1.3 Key-Rotation und Verwaltung

Statische API-Keys, die jahrelang gültig sind, sind ein Sicherheitsrisiko. Setze stattdessen auf regelmäßige Rotation (alle 30–90 Tage) und speichere Keys gehasht in der Datenbank – niemals im Klartext. Tools wie Vault by HashiCorp oder Infisical helfen bei der zentralen Secrets-Verwaltung.

2. Rate-Limiting – Schutz vor Überlastung und Missbrauch

Rate-Limiting ist die wichtigste Maßnahme, um deine LLM-Infrastruktur vor Missbrauch zu schützen. Ohne Begrenzung kann ein einzelner Angreifer tausende Anfragen pro Sekunde senden, deine GPU-Auslastung in die Höhe treiben und enorme Kosten verursachen.

2.1 Nginx Rate-Limiting (ngx_http_limit_req_module)

Nginx bietet mit limit_req ein leistungsstarkes Rate-Limiting auf Basis der Leaky-Bucket-Algorithmus-Variante:

# /etc/nginx/nginx.conf – im http-Block
limit_req_zone $binary_remote_addr zone=llm:10m rate=5r/s;
limit_req_zone $http_x_api_key zone=llm_by_key:10m rate=10r/s;

# Im Server-Block deiner LLM-API
location /v1/chat/completions {
    # Pro IP: max 5 Requests pro Sekunde, Burst bis 10
    limit_req zone=llm burst=10 nodelay;

    # Pro API-Key: max 10 Requests pro Sekunde
    limit_req zone=llm_by_key burst=20 nodelay;

    proxy_pass http://llm_backend;
}

Die erste Zone (llm) limitiert pro IP-Adresse – das schützt vor einfachen DoS-Angriffen. Die zweite Zone (llm_by_key) limitiert pro API-Key – so können legitime Nutzer hinter einer NAT-IP nicht blockiert werden.

2.2 Cloudflare Rate Limiting (Managed + Custom)

Betreibst du deine LLM-API hinter Cloudflare, profitierst du vom integrierten Rate Limiting auf CDN-Ebene. Vorteil: Angreifer erreichen deinen Server gar nicht erst. So konfigurierst du es:

  1. Im Cloudflare-Dashboard unter Security → WAF → Rate Limiting Rules
  2. Regel erstellen: http.request.uri.path contains "/v1/chat/completions"
  3. Limit setzen: 100 Requests pro Minute pro Besucher
  4. Aktion: Block oder Managed Challenge (für legitime User)
  5. Optional: Super Bot Fight Mode aktivieren, um Scraper und Bots automatisch zu blockieren

Wir empfehlen eine Kombination aus Cloudflare Rate Limiting (Layer-7) und Nginx Rate Limiting (Server-Ebene) für eine Defense-in-Depth-Strategie.

Methode Ebene Vorteile Nachteile
Nginx limit_req Application (Server) Kostengünstig, flexibel konfigurierbar, per IP/Key/Header Schützt nicht vor großvolumigen DDoS – Server-Ressourcen werden trotzdem belastet
Cloudflare Rate Limiting CDN / Edge Angreifer erreicht Server nicht, Challenge-Mechanismus, einfache Konfiguration Nur mit Cloudflare-Proxy, kostenpflichtig für höhere Limits (Pro/ Business)
Fail2ban OS / Firewall Blockiert IPs dauerhaft nach N Fehlversuchen, iptables/nftables-Integration Reaktiv – schützt nicht vor Burst-Angriffen, Log-Analyse benötigt
API-Gateway (Kong etc.) Service Mesh Zentrales Rate-Limiting für mehrere LLM-Backends, Redis-Cluster für verteilte Limits Höhere Komplexität, zusätzliche Infrastruktur-Kosten

3. Request-Logging – Sichtbarkeit als Sicherheitsfaktor

Ohne Logging fliegen Missbrauch, Fehlkonfigurationen und Angriffe unter dem Radar. Ein strukturiertes Logging-System ist die Grundlage für Security-Incident-Response, Forensik und Kosten-Tracking.

3.1 Nginx Access Logs mit JSON-Format

Statt des Standard-Nginx-Log-Formats verwendest du JSON – so lassen sich Logs mit Elasticsearch, Loki oder Splunk maschinell auswerten:

# Nginx-Logformat definieren
log_format llm_json escape=json
  '{'
    '"time": "$time_iso8601",'
    '"remote_addr": "$remote_addr",'
    '"request_method": "$request_method",'
    '"request_uri": "$request_uri",'
    '"status": $status,'
    '"body_bytes_sent": $body_bytes_sent,'
    '"request_time": $request_time,'
    '"http_referer": "$http_referer",'
    '"http_user_agent": "$http_user_agent",'
    '"api_key_prefix": "$http_authorization"'
  '}';

access_log /var/log/nginx/llm-api-access.log llm_json;

⚠️ Wichtiger Hinweis: Logge niemals vollständige API-Keys im Klartext. Verwende stattdessen nur den Präfix (erste 4–8 Zeichen) zur Identifikation, wie im Beispiel oben. Auch die Logs der LLM-Modelle selbst – also die Prompts und Responses – gehören nicht in Standard-Access-Logs. Trenne technische Logs (Metadaten) von Inhalts-Logs (Audit-Trail mit spezieller Zugriffskontrolle).

3.2 Log-Aggregation mit Loki oder Elasticsearch

Für die Auswertung empfehlen wir eine zentrale Log-Plattform:

  • Grafana Loki – Leichtgewichtig, ideal für VPS-Betrieb (Setup-Guide), Promtail als Log-Shipper
  • ELK Stack (Elasticsearch, Logstash, Kibana) – Volle Volltext-Suche, Visualization, aber ressourcenintensiver
  • Vector (Datadog Agent) – Moderner, ressourcenschonender Log-Shipper mit Lua-Transformationen

4. Modell-Isolation – Mehrere LLMs sicher betreiben

Betreibst du mehrere Modelle auf einem Server – etwa ein günstiges 7B-Modell für öffentliche Chatbots und ein leistungsstarkes 70B-Modell für interne Analysen – müssen diese sauber voneinander isoliert sein.

4.1 Container-Isolation mit Docker

Der einfachste Weg: Jedes Modell bzw. jeder LLM-Dienst läuft in einem eigenen Docker-Container mit dedizierten Ressourcen-Limits:

# docker-compose.yml
services:
  ollama-public:
    image: ollama/ollama:latest
    deploy:
      resources:
        limits:
          cpus: '4'
          memory: 16G
    environment:
      - OLLAMA_HOST=0.0.0.0:11434
    networks:
      - llm-internal
    volumes:
      - ollama-data:/root/.ollama

  ollama-internal:
    image: ollama/ollama:latest
    deploy:
      resources:
        limits:
          cpus: '8'
          memory: 64G
    environment:
      - OLLAMA_HOST=0.0.0.0:11435
    networks:
      - llm-internal
      - vpn-only
    volumes:
      - ollama-internal-data:/root/.ollama

  nginx-gateway:
    image: nginx:alpine
    ports:
      - "443:443"
    volumes:
      - ./nginx/:/etc/nginx/conf.d/
    networks:
      - llm-internal

Das öffentliche Modell hat nur Zugriff auf das llm-internal-Netzwerk, während das interne Modell zusätzlich im vpn-only-Netzwerk hängt und nur über WireGuard erreichbar ist.

4.2 Ressourcen-Isolation mit cgroups und systemd

Ohne Docker steuerst du Ressourcen-Limits direkt über systemd-Service-Units:

[Service]
ExecStart=/usr/local/bin/ollama serve
CPUQuota=400%
MemoryMax=16G
MemorySwapMax=0
IOWeight=100

So verhindert du, dass ein fehlerhaftes oder missbräuchliches Modell den gesamten Server lahmlegt.

5. CORS-Konfiguration – Wer darf deine API aufrufen?

Wenn deine LLM-API aus dem Browser heraus angesprochen wird (z. B. von einer React-App oder Open WebUI), greifen Cross-Origin Resource Sharing (CORS)-Richtlinien. Eine zu lasche CORS-Konfiguration erlaubt jeder Webseite, Requests von fremden Domains an deine API zu senden.

So konfigurierst du CORS sicher in Nginx:

location /v1/ {
    # Nur erlaubte Origins – NIEMALS * verwenden bei API-Keys!
    set $cors_origin "";
    if ($http_origin ~ "^(https?://meine-webui\.de|https?://app\.eigenes-team\.com)$") {
        set $cors_origin $http_origin;
    }

    add_header Access-Control-Allow-Origin "$cors_origin" always;
    add_header Access-Control-Allow-Methods "GET, POST, OPTIONS" always;
    add_header Access-Control-Allow-Headers "Authorization, Content-Type, X-Request-ID" always;
    add_header Access-Control-Max-Age 86400;

    # Preflight (OPTIONS) direkt beantworten
    if ($request_method = OPTIONS) {
        add_header Content-Length 0;
        add_header Content-Type text/plain;
        return 204;
    }

    proxy_pass http://llm_backend;
}

Faustregel: Erlaube Access-Control-Allow-Origin niemals *, wenn Authentifizierung (API-Key) im Spiel ist. Setze explizit die tatsächlich genutzten Domains oder verwende eine dynamische Prüfung über eine Whitelist.

6. DDoS-Schutz für LLM-APIs

LLM-APIs sind ein attraktives Ziel für DDoS-Angriffe, da ein einzelner Request GPU-Zeit verbraucht und damit teuer ist. Ein sogenannter LLM-Pricing-Angriff (auch LLM API Cost Attack genannt) zielt darauf ab, durch massenhafte Anfragen mit maximaler Token-Länge deine Rechnung in die Höhe zu treiben.

6.1 Cloudflare DDoS-Schutz

Cloudflare bietet integrierten DDoS-Schutz – auch im Free-Tarif:

  • Layer 3/4: Automatische Netzwerk-DDoS-Mitigation (SYN-Flood, UDP-Amplification)
  • Layer 7: WAF-Regeln gegen HTTP-Floods, Rate Limiting (siehe Abschnitt 2.2)
  • Bot Management: Super Bot Fight Mode blockiert bekannte und mutmaßliche Bots
  • Security Level: Bei Angriffen automatisch auf "I'm Under Attack!" umschalten

6.2 Server-Level-Schutz

Ergänzend zu Cloudflare schützt du den Server selbst:

  • Fail2ban – Blockiert IPs nach N 401/403-Fehlern innerhalb eines Zeitfensters
  • Nginx liMit – limit_conn für parallele Verbindungen, limit_req für Request-Rate
  • Kernel-Tuning – net.core.somaxconn, net.ipv4.tcp_syncookies, nf_conntrack_max anpassen
  • Max Body Size – client_max_body_size 10m verhindert oversized Requests (extrem lange Prompts)
# Weitere Nginx-DDoS-Schutz-Maßnahmen
client_body_timeout 10s;
client_header_timeout 10s;
send_timeout 10s;
client_max_body_size 10m;
limit_conn per_ip 5;
limit_conn_zone $binary_remote_addr zone=per_ip:10m;

7. Sicherheits-Checkliste für deine LLM-API

Zum Abschluss eine kompakte Checkliste, mit der du deine KI-Server-Infrastruktur systematisch überprüfen kannst:

  • 🔑 Authentifizierung: API-Key-Pflicht für jeden Endpunkt (außer Health-Checks) – Kritisch
  • 🔒 HTTPS: Let's Encrypt mit Auto-Renewal, HSTS-Header (max-age=31536000; includeSubDomains) – Kritisch
  • 🚦 Rate Limiting: Pro IP + pro API-Key, unterschiedliche Limits für verschiedene Endpunkte – Hoch
  • 📝 Logging: JSON-Logs mit Loki/ELK, API-Key-Präfix statt Voll-Key, Monats-Retention – Hoch
  • 🧩 Modell-Isolation: Container-Trennung, Ressourcen-Limits (CPU/RAM) pro Modell – Hoch
  • 🌐 CORS: Explizite Whitelist statt *, Preflight-Handling – Hoch
  • 🛡️ DDoS-Schutz: Cloudflare vorgeschaltet, Fail2ban aktiv, Nginx-Limits gesetzt – Hoch
  • 🔐 Netzwerk: API nur hinter VPN (WireGuard) oder Reverse Proxy, kein direkter Port-Zugriff – Kritisch
  • 🔄 Updates: Automatische Sicherheits-Updates für Docker, Nginx, LLM-Tools – Hoch
  • 📊 Monitoring: Prometheus-Metriken für Request-Rate, Latenz, Fehlerraten, GPU-Auslastung – Mittel

Fazit: Sicherheit ist kein Feature, sondern ein Prozess

Die Absicherung einer selbst gehosteten LLM-API ist komplexer als das Setup eines einfachen Web-Servers, aber mit den richtigen Werkzeugen gut beherrschbar. Die Kombination aus API-Key-Authentifizierung, mehrstufigem Rate-Limiting (Edge + Server), strukturiertem Request-Logging und Container-basierter Modell-Isolation bildet ein robustes Sicherheitsfundament. Ergänzt durch Cloudflare als DDoS-Schutz und restriktive CORS-Richtlinien schützt du deine KI-Infrastruktur zuverlässig vor den häufigsten Angriffsvektoren.

Denk daran: Sicherheit ist kein einmaliges Projekt, sondern ein fortlaufender Prozess. Führe regelmäßige Security-Reviews durch, rotiere API-Keys, überwache deine Logs auf Anomalien und halte deine Software auf dem aktuellen Stand. Wer diese Grundlagen beachtet, kann die Vorteile von Self-Hosted-LLMs – Datenhoheit, Kostentransparenz und volle Kontrolle – ohne Sicherheitskompromisse genießen.
