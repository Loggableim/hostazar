Docker-Container für LLM-Inference – Best Practices 2026

GPU-Passthrough, Ollama, vLLM und Open WebUI in Docker Compose: Wie du Large Language Models performant, sicher und wartbar in Containern betreibst.

— Anzeige —

Google AdSense Platzhalter

Large Language Models (LLMs) wie Llama 3.3, Mistral oder Qwen 2.5 in Docker-Containern zu betreiben, ist 2026 zum Standard geworden. Container vereinfachen das Deployment, isolieren Abhängigkeiten und ermöglichen reproduzierbare Setups – sei es auf einem einzelnen GPU-VPS, einem dedizierten Server im Rechenzentrum oder einer Multi-Node-Kubernetes-Umgebung.

Doch LLM-Inference stellt besondere Anforderungen an Container: GPUs müssen durchgereicht werden, Modelle wiegen 10–80 GB, der Speicherbedarf ist enorm und die Netzwerklatenz zwischen Container und GPU-Treiber darf nicht zum Flaschenhals werden. In diesem Guide zeigen wir dir die bewährten Praktiken für Docker-Container mit LLM-Inference – von der Basis mit dem NVIDIA Container Toolkit über Ollama und vLLM bis zum vollständigen Docker-Compose-Setup mit Open WebUI und einer Datenbank.

Warum Docker für LLM-Inference?

Container bieten für KI-Workloads entscheidende Vorteile: Du kapselst CUDA-Treiber, Python-Runtime und Modell-Dateien in einem Image, das auf jedem Docker-Host mit NVIDIA-GPU identisch läuft. Versionierung, Rollbacks und parallele Instanzen verschiedener Modelle werden trivial. Statt auf dem Hostsystem eine komplexe CUDA-Umgebung mit mehreren Python-Versionen und Konflikten zu pflegen, deklarierst du alles im Dockerfile oder Compose-Service.

Hinzu kommt die Ressourcenisolierung: Mit Docker limitierst du CPU, RAM und GPU-Speicher pro Container. Ein fehlerhaftes Modell kann dadurch nicht den gesamten Server lahmlegen. Für Teams, die mehrere Modelle parallel betreiben oder eine LLM-Infrastruktur mit CI/CD aufbauen, ist Docker der logische nächste Schritt nach dem manuellen Setup. Für eine grundlegende Einführung in Docker Compose empfehlen wir unseren Docker-Compose-VPS-Guide.

NVIDIA Container Toolkit – die Basis für GPU-Passthrough

Damit ein Docker-Container auf die GPU zugreifen kann, reicht ein einfaches --gpus all nicht mehr aus – zumindest nicht, wenn du eine saubere, zukunftssichere Konfiguration möchtest. Das NVIDIA Container Toolkit (ehemals nvidia-docker2) ist der offizielle Weg, um CUDA-fähige Container zu betreiben. Es stellt das NVIDIA-Runtime-Environment für Docker bereit und sorgt dafür, dass die GPU-Treiber, CUDA-Bibliotheken und der Container-Kernel-Modul-Zugriff korrekt gekapselt werden.

Installation des NVIDIA Container Toolkits

Die Installation erfolgt auf Ubuntu 24.04 LTS in wenigen Schritten:

curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
  sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg

curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list

sudo apt update && sudo apt install -y nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker

Nach einem Neustart von Docker prüfst du die GPU-Erreichbarkeit mit einem ersten CUDA-Container:

docker run --rm --gpus all nvidia/cuda:12.6-base-ubuntu24.04 nvidia-smi

Wenn nvidia-smi die GPU anzeigt, ist der GPU-Passthrough einsatzbereit. Ab Docker 25+ und neueren Versionen des Toolkits ist die NVIDIA-Container-Runtime automatisch als Default-Runtime konfigurierbar – du musst dann nicht bei jedem Container --gpus all angeben.

Ollama im Docker-Container

Ollama bietet ein offizielles Docker-Image, das NVIDIA-GPUs out of the box unterstützt. Der Start ist denkbar einfach:

docker run -d --gpus all \
  -v ollama_data:/root/.ollama \
  -p 11434:11434 \
  --name ollama \
  ollama/ollama

Das Volume ollama_data speichert heruntergeladene Modelle dauerhaft. Nach dem Start lädst du ein Modell in den Container:

docker exec ollama ollama pull llama3.3:70b-instruct-q4_K_M

Für Produktivumgebungen solltest du die Ollama-Umgebungsvariablen OLLAMA_NUM_PARALLEL (Anzahl paralleler Requests), OLLAMA_MAX_LOADED_MODELS und OLLAMA_KEEP_ALIVE setzen, um das Verhalten an deine Hardware anzupassen. Ein vollständiger Aufruf mit Konfiguration:

docker run -d --gpus all \
  -v ollama_data:/root/.ollama \
  -p 11434:11434 \
  -e OLLAMA_NUM_PARALLEL=4 \
  -e OLLAMA_MAX_LOADED_MODELS=2 \
  -e OLLAMA_KEEP_ALIVE=5m \
  --memory="64g" \
  --memory-swap="64g" \
  --shm-size=8g \
  --name ollama \
  --restart unless-stopped \
  ollama/ollama

Die vollständige Anleitung zum Betrieb von Ollama auf einem VPS findest du in unserem Artikel Ollama LLM Server auf VPS hosten.

vLLM: High-Performance-Inference für produktive Setups

Während Ollama auf Einfachheit und schnelle Experimente ausgelegt ist, zielt vLLM auf maximale Inferenz-Performance in Produktionsumgebungen. vLLM implementiert PagedAttention, continuous batching und optimierte CUDA-Kernel, die den GPU-Durchsatz um ein Vielfaches steigern können. Besonders bei vielen parallelen Requests – zum Beispiel in einer Chat-API mit mehreren Nutzern – glänzt vLLM.

Der offizielle vLLM-Docker-Container wird über das NVIDIA CUDA-Base-Image bereitgestellt und unterstützt ebenfalls GPU-Passthrough:

docker run -d --gpus all \
  -v /models:/models \
  -p 8000:8000 \
  --name vllm \
  vllm/vllm-openai:latest \
  --model /models/Meta-Llama-3.3-70B-Instruct-Q4_K_M \
  --tensor-parallel-size 2 \
  --gpu-memory-utilization 0.9 \
  --max-model-len 8192 \
  --enforce-eager

Mit --tensor-parallel-size 2 verteilst du das Modell auf zwei GPUs. Der Parameter --gpu-memory-utilization 0.9 reserviert 90 % des VRAMs für das Modell – der Rest bleibt für KV-Cache und Overhead. vLLM startet automatisch eine OpenAI-kompatible API auf Port 8000, sodass du sie mit Open WebUI, Continue.dev oder LangChain verbinden kannst.

Docker Compose: Multi-Container-Setup mit LLM + Open WebUI + DB

Ein vollständiges LLM-Inference-Setup besteht oft aus mehreren Komponenten: der Inference-Engine (Ollama oder vLLM), einem Chat-Frontend (Open WebUI) und optional einer Datenbank für Chat-Verläufe und Nutzerdaten. Mit Docker Compose definierst du alle Dienste in einer einzigen YAML-Datei und startest sie mit einem Befehl.

Beispiel: Ollama + Open WebUI + PostgreSQL

services:
  ollama:
    image: ollama/ollama:latest
    container_name: ollama
    restart: unless-stopped
    volumes:
      - ollama_data:/root/.ollama
    ports:
      - "11434:11434"
    environment:
      - OLLAMA_NUM_PARALLEL=4
      - OLLAMA_KEEP_ALIVE=5m
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: all
              capabilities: [gpu]
    networks:
      - llm-net

  open-webui:
    image: ghcr.io/open-webui/open-webui:main
    container_name: open-webui
    restart: unless-stopped
    ports:
      - "8080:8080"
    volumes:
      - open-webui_data:/app/backend/data
    environment:
      - OLLAMA_BASE_URL=http://ollama:11434
      - WEBUI_SECRET_KEY=${WEBUI_SECRET_KEY}
    depends_on:
      - ollama
    networks:
      - llm-net

  postgres:
    image: postgres:16-alpine
    container_name: llm-db
    restart: unless-stopped
    volumes:
      - postgres_data:/var/lib/postgresql/data
    environment:
      POSTGRES_DB: webui
      POSTGRES_USER: webui
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    networks:
      - llm-net

volumes:
  ollama_data:
  open-webui_data:
  postgres_data:

networks:
  llm-net:
    driver: bridge

Dieses Setup trennt die Dienste sauber: Ollama läuft mit GPU-Passthrough, Open WebUI verbindet sich über das interne Docker-Netzwerk (llm-net) und PostgreSQL speichert die Chat-Verläufe. Die Zugangsdaten landen in einer .env-Datei, nicht in der Compose-Datei selbst. Für den produktiven Betrieb empfehlen wir zusätzlich einen Reverse Proxy (Traefik oder Nginx) vor Open WebUI, wie in unserem Docker-Compose-VPS-Guide beschrieben.

Ressourcenlimits und GPU-Monitoring

LLM-Container sind ressourcenhungrig. Ohne Limits kann ein einzelner Container den gesamten Server blockieren. Setze daher immer explizite CPU- und RAM-Grenzen in der Compose-Datei:

deploy:
  resources:
    limits:
      cpus: '8'
      memory: 64G
    reservations:
      cpus: '4'
      memory: 32G
      devices:
        - driver: nvidia
          count: 1
          capabilities: [gpu]

Für das GPU-Monitoring empfiehlt sich dcgm-exporter (NVIDIA Data Center GPU Manager) in einem separaten Container. Er exportiert Metriken wie GPU-Auslastung, Temperatur, VRAM-Verbrauch und Power Draw, die du mit Prometheus und Grafana visualisieren kannst. Eine einfachere Alternative ist nvidia-smi im Host-System mit regemäßigem Logging via Cronjob.

Auch der Speicherplatz wächst schnell: Ein einziges 70B-Modell in Q4 belegt ~40 GB, mehrere Modelle plus Docker-Images und Build-Cache können 200–500 GB füllen. Richte daher eine regelmäßige Bereinigung mit docker system prune -f und ollama prune ein. Für passende SSDs und Server-Hardware findest du Empfehlungen bei Amazon*.

Sicherheit: GPU-Container absichern

Container mit GPU-Zugriff haben ein erweitertes Privilegienprofil. Der Zugriff auf /dev/nvidia* und /dev/dri* gibt dem Container theoretisch direkten Hardware-Zugriff. Beachte daher folgende Sicherheitsmaßnahmen:

  • Kein --privileged: Für GPU-Passthrough ist --privileged nicht nötig. Nutze stattdessen explizite Device-Reservations wie im Compose-Beispiel oben.
  • Read-Only Root Filesystem: Wenn das Modell auf einem externen Volume liegt, kannst du den Container-RO-FS mit --read-only aktivieren.
  • Keine öffentlichen Ports: Inference-APIs (Port 11434, 8000) niemals direkt exposen. Setze einen Reverse Proxy mit Authentifizierung davor oder nutze ein VPN (WireGuard).
  • Regelmäßige Updates: Halte Docker Engine, NVIDIA-Treiber und das Container Toolkit aktuell. Sicherheitslücken wie CVE-2024-在半 (Container Escape via GPU) wurden mehrfach in NVIDIA-Treibern gefunden.
  • User Namespaces: Aktiviere userns-remap in der Docker-Daemon-Konfiguration, um Container-Prozesse mit eingeschränkten Rechten laufen zu lassen.

Best Practices für das Docker-Image

Wie bei jedem Docker-Workload gilt: Das Image sollte so klein und sicher wie möglich sein. Für LLM-Inference bedeutet das konkret:

  • CUDA-Base-Image statt vollständigem OS: Starte von nvidia/cuda:12.6-runtime-ubuntu24.04 statt ubuntu:24.04 – das spart 2–3 GB Image-Größe und enthält bereits die optimierten CUDA-Bibliotheken.
  • Multi-Stage-Builds: Baue Abhängigkeiten in einer Build-Stage mit vollständigem CUDA-Devel-Image und kopiere nur die Binary und Laufzeitbibliotheken in das finale Runtime-Image.
  • Modell nicht im Image: Lade Modelle zur Laufzeit oder mounte sie als Volume. Ein Image mit 40-GB-Modell-Gewichten ist impractical für CI/CD und Deployment-Pipelines.
  • Healthcheck integrieren: Definiere einen Docker-Healthcheck, der regelmäßig die API erreichbarkeit prüft. Bei Ollama: HEALTHCHECK CMD ollama ps || exit 1.
  • Non-Root-User: Auch wenn CUDA-Container oft als Root laufen – viele Images wie das offizielle vLLM-Image unterstützen bereits den vllm-User. Passe die Berechtigungen für die Modell-Volumes entsprechend an.

🚀 Praxis-Tipp: Starte mit Ollama im Docker-Container für erste Experimente. Sobald du mehrere Nutzer bedienen oder eine Produktions-API bereitstellen möchtest, migriere zu vLLM. Lege für jedes Setup ein eigenes Projektverzeichnis mit compose.yml, .env und einem kurzen README.md an – so behältst du auch nach Monaten den Überblick.

Fehlerbehebung: Typische Probleme bei LLM-Containern

  • CUDA out of memory: Reduziere gpu-memory-utilization oder wähle ein kleineres Modell / stärkere Quantisierung. Prüfe mit nvidia-smi den aktuellen VRAM-Verbrauch.
  • Container startet keine GPU: Überprüfe docker info | grep -i runtime. Fehlt die NVIDIA-Runtime, installiere das Container Toolkit neu und starte Docker neu.
  • Langsame Modell-Ladezeiten: Nutze eine NVMe-SSD statt SATA-SSD. Die Modell-Gewichte müssen vor der ersten Inference komplett in den VRAM geladen werden – bei 40 GB dauert das auf SATA schnell 30–60 Sekunden.
  • Open WebUI verbindet sich nicht mit Ollama: Stelle sicher, dass OLLAMA_BASE_URL auf den Container-Namen zeigt (http://ollama:11434) und beide Container im selben Docker-Netzwerk sind.
  • Netzwerk-Timeout bei langen Prompts: Erhöhe --shm-size (Shared Memory) im Container – einige Inference-Engines nutzen Shared Memory für den KV-Cache.

Fazit

Docker-Container sind 2026 der Standard für den Betrieb von LLM-Inference-Workloads. Das NVIDIA Container Toolkit liefert die GPU-Passthrough-Basis, Ollama und vLLM decken die gesamte Bandbreite von schnellen Experimenten bis zur High-Performance-Produktions-API ab, und Docker Compose orchestriert das Multi-Container-Setup aus Inference-Engine, Frontend und Datenbank.

Der Schlüssel zu einem stabilen Betrieb liegt in drei Punkten: saubere Ressourcenlimits, durchdachte Netzwerktrennung und regelmäßige Updates von Treibern und Containern. Mit den hier gezeigten Best Practices – GPU-Reservierung statt Privileged-Mode, Read-Only-Volumes, keine offenen Ports, Healthchecks und NVMe-SSDs – stellst du sicher, dass deine LLM-Container zuverlässig, schnell und sicher laufen.
